# 노드-엣지 워크플로 IDE — 완전 개발 플랜

> 함수로 잇는 대신, 각자가 완결된 실행 주기를 가진 단위(노드)를 5W1H 선언형 프로토콜로 잇는 시각적 프로그래밍 IDE. Tauri 기반. 에이전트는 노드를 *생성*하고, 인간은 노드를 *배선*한다.

---

## 0. 이 문서의 읽는 법

각 Phase는 다음 네 블록으로 구성된다.

- **목표** — 이 Phase가 끝났을 때 존재해야 하는 것 (산출물 정의).
- **스텝** — 순서 있는 구현 단위. 각 스텝은 검증 가능한 결과물을 남긴다.
- **리스크** — 이 단계에서 실제로 깨지거나 잘못 결정되기 쉬운 지점.
- **의존** — *선행*(이 Phase가 의존하는 앞 Phase)과 *후행*(이 Phase에 의존하는 뒤 Phase). 후행 의존이 곧 이 Phase를 먼저 짓는 이유다.

Phase는 7개 Part로 묶인다. Part는 권장 작업 순서이지 엄격한 직렬은 아니다. Part I·II는 반드시 선행해야 한다(전부가 여기 의존). Part III~VII는 의존 그래프가 허락하는 범위에서 병렬 진행 가능.

---

## 1. 불변식 (Cross-Cutting Invariants)

모든 Phase에서 깨지면 안 되는 설계 척추. 어떤 구현 결정도 이 11개와 충돌하면 그 결정이 틀린 것이다.

1. **일급 시민은 노드 하나뿐.** adapter·gate·throttle·transform·trigger 전부 노드다. 변환은 특별 지위를 갖지 않는다. (대수적 폐포성)
2. **그래프는 거짓말하지 않는다.** 인간이 그래프 층에서 신뢰하는 것은 노드의 *선언*이며, 선언과 실제 행동의 일치는 런타임 가드가 강제한다. 임의 언어 노드의 선언은 *증명이 아니라 약속*이므로 경계에서 검증된다.
3. **경계 = blast radius = knowledge radius.** 노드 격리는 안전(영향 차단)·재사용(맥락 없이 신뢰)·에이전트 봉쇄·stateless 생성을 *하나의 경계 개념*으로 동시에 보장한다.
4. **5W1H 축은 직교한다.** 각 원칙(What/How/When/Where/Who/Why)에 4개 프리셋 + `custom`. 두 노드의 상호작용은 양쪽 선언의 곱으로 *결정론적으로 유도*된다(연결 시점에 사람이 정할 것이 없다).
5. **해소 격자에 빈칸은 없다.** 모든 프리셋 쌍은 {자동호환 | 어댑터삽입 | 거부} 중 하나로 정의된다. 미정의 칸의 기본값은 "조용한 통과"가 아니라 **거부**다.
6. **구조는 엄격, 데이터타입은 점진.** 구조 컴파일(포트 연결 정합성, 루프 안전장치, 리소스 선언)은 strict-reject. 데이터타입 호환은 soft/gradual + 런타임 가드.
7. **backpressure는 구조적이다.** 모든 커넥션은 bounded buffer를 갖는다. 버퍼가 차면 상류가 막힌다. 수동 throttle 노드는 보조 수단일 뿐 기본 안전장치가 아니다.
8. **생성과 권한은 분리된다.** 에이전트는 노드를 생성한다. 인간은 토폴로지(배선)를 결정한다. 그래프는 인간 검토에 최적화된 산출물이다.
9. **서사와 산물은 분리된다.** 노드를 만든 *명세(5W1H 선언 + 요청)*는 노드에 인라인으로 박힌다(재생성용). 개발 *세션 기록*은 ID로 외부 참조된다(회상/"왜?" 질의용).
10. **에이전트는 stateless로 스폰된다.** 노드 생성 작업은 서로 독립적이므로 과거 기억이 불필요하다. 따라서 병렬화는 공짜다(격리할 공유 상태가 없다).
11. **에이전트 구동은 구조화 스트리밍으로.** 터미널 스크레이프(scrollback 파싱)는 금지. 에이전트 노드는 headless/JSON I/O로 구동되어, 인간에겐 렌더링된 터미널 뷰를, 워크플로엔 깨끗한 구조화 메시지를 동시에 제공한다.

## 2. 의존 그래프 개요

```
Part I  (프로토콜)        Phase 1 → 2 → 3 → 4         [전부의 토대]
                              │
Part II (런타임 커널)      Phase 5 → 6 → 7 → 8
                              │
        ┌─────────────────────┼─────────────────────┐
Part III(노드 호스트)     9   10   11   12            [병렬 가능, 8에 의존]
        └─────────────────────┼─────────────────────┘
                              │
Part IV (합성/컴파일)     13 → 14 → 15 → 16
                              │
Part V  (Tauri IDE)       17 → 18 → 19 → 20 → 21       [16·11에 의존]
                              │
Part VI (에이전트 저작)    22 → 23                      [2·11·21에 의존]
                              │
Part VII(하드닝/운영)     24 25 26 27 28 29            [병렬, 각자 선행 상이]
```

핵심 임계 경로(critical path): **1 → 2 → 3 → 5 → 6 → 7 → 13 → 16 → 17 → 18**. 이 사슬이 "노드를 만들어 그래프로 잇고 컴파일해 IDE에서 돌린다"는 최소 완성품(M3, §아래 마일스톤)을 직접 결정한다. 나머지는 이 척추에 붙는 살이다.

# Part I — 형식적 토대

> 코드보다 정의가 먼저다. 여기서 잘못 그은 선은 Part V에서 수백 배로 청구된다. 이 Part의 산출물은 대부분 스펙 문서 + 순수 참조 구현(UI·런타임 무관)이다.

---

## Phase 1 — 5W1H 상호작용 프로토콜 형식화

**목표.** 6개 직교 축과 각 축의 4개 프리셋(+`custom`)을 *기계가 읽을 수 있는 형식 명세*로 확정한다. 이 문서가 이후 모든 코드의 단일 진실 원천(SSOT)이 된다.

**스텝.**
1. 6개 축의 의미론을 한 문장씩 비모호하게 정의한다. (What=무엇이 흐르나, How=어떻게 흐르나, When=언제 발화하나, Where=어디로/포트 토폴로지, Who=누구와 호환되나, Why=어떤 조건/의존에서 흐르나.)
2. 축별 4개 프리셋을 열거하고 각각의 정확한 런타임 의미를 명시한다. (예: How = `single` / `stream` / `request-response` / `broadcast`; When = `once` / `on-trigger` / `persistent` / `per-iteration`.)
3. 각 축에 `custom` 프리셋을 정의한다 — 인라인 코드 탈출구가 들어가는 자리이자, 프리셋으로 표현 불가한 경우의 escape.
4. 프리셋을 식별하는 안정적 식별자 체계(버저닝 포함)를 정한다. 프리셋 추가는 가능하되 *제거·의미 변경은 금지*(하위호환 규칙)임을 명문화한다.
5. 6축 선언의 직렬화 스키마(JSON Schema 등)를 작성하고, 유효/무효 예제 세트를 첨부한다.
6. "직교성 자가 점검" 절차를 정의한다 — 한 축의 프리셋 선택이 다른 축의 의미를 바꾸면 직교성이 깨진 것이므로, 그 경우를 검출하는 체크리스트.

**리스크.**
- 축이 사실은 직교가 아닐 수 있다(예: How=`stream`이 When=`persistent`를 암묵 강제). 이러면 격자가 폭발한다. → 각 프리셋 쌍을 손으로 교차 검증해 숨은 결합을 색출.
- 프리셋 4개가 "너무 적어" 흔한 케이스를 못 덮을 수 있다. → 의도적으로 4로 닫되, 못 덮는 건 전부 `custom`+변환 노드로 흘려보낸다는 원칙을 여기서 확정(Phase 15가 이를 떠받침).
- 의미를 나중에 "조금만" 바꾸고 싶은 유혹. 한 번 배포된 프리셋 의미 변경은 이미 만든 모든 노드를 깨뜨린다. → 변경 금지를 불변식으로 못 박음.

**의존.** 선행: 없음(루트). 후행: **거의 전부.** 특히 Phase 2(매니페스트), 3(해소 격자), 4(와이어 포맷), 13(구조 컴파일). 이 명세가 흔들리면 전 시스템이 흔들린다.

## Phase 2 — 노드 매니페스트 & SDK 계약 확정

**목표.** 노드가 자신을 선언하는 매니페스트 포맷과, 노드 저자(인간 또는 에이전트)가 따르는 node-sdk 계약을 형식화한다. *기능을 노드에 격리*하고 *전체 이해 없이 한 노드만 만들면 끝*인 상태를 가능케 하는 표면.

**스텝.**
1. 매니페스트 필수 필드를 정의한다: 노드 정체성(이름/버전/식별자), 포트 목록(각 포트의 6축 선언), 라이프사이클 선언(persistent/ephemeral), 리소스 요구(네트워크/파일시스템/시크릿), 그리고 **프로비넌스 슬롯**(생성 세션 ID + 인라인 명세, 불변식 9).
2. node-sdk가 노드 저자에게 제공하는 인터페이스를 설계한다: 입력 수신 훅, 출력 발신 핸들, 라이프사이클 콜백(init/run/stop/health), 에러 보고 채널.
3. SDK의 *기본값(default)*을 정의한다 — 최소 노드는 `any` I/O + no-op 라이프사이클을 공짜 상속해 "한 줄로" 만들어진다(저작 마찰 최소화).
4. 언어 무관성을 위한 두 계층을 분리 정의한다: (a) in-process 타입드 노드(증명 가능), (b) out-of-process 외부 실행 노드(약속 기반, Phase 9가 구현).
5. 매니페스트 검증기(linter)를 순수 함수로 구현한다 — 매니페스트만 보고 구조적 유효성을 판정(런타임 불필요).
6. "내 스크립트를 노드로" 만드는 최소 래퍼 예제를 3개 언어(예: Python/TS/Rust)로 작성해 SDK 계약이 실제로 언어 무관함을 증명한다.

**리스크.**
- 매니페스트가 비대해지면 노드 저작 마찰이 커져 불변식 3·10의 이점(에이전트가 쉽게 만든다)이 희석된다. → 필수 필드를 최소화하고 나머지는 default 상속.
- in-process와 out-of-process 노드의 계약을 하나로 뭉치면 "약속을 증명인 척"하게 된다(불변식 2 위반). → 두 계층을 매니페스트 레벨에서 명시적으로 구분.

**의존.** 선행: Phase 1. 후행: 9·10·11·12(모든 호스트가 이 계약 구현), 16(컴파일러가 매니페스트 소비), 22(에이전트가 이 SDK로 노드 생성), 23(프로비넌스), 28(패키징).

## Phase 3 — 프리셋 해소 격자 설계

**목표.** 임의의 두 노드를 이을 때 *상호작용을 자동 유도*하는 해소 규칙(resolution matrix)을 정의·구현한다. 불변식 4·5의 심장.

**스텝.**
1. 각 축마다 4×4(=16) 프리셋 쌍에 대해 결과를 {자동호환 | 어댑터삽입 | 거부}로 명시한다. 6축이면 96칸 + `custom` 행/열.
2. "어댑터삽입" 칸마다 *어떤 변환 노드*가 끼는지를 지정한다(예: How `stream`→`single`은 first-value 어댑터인지 buffer-collect 어댑터인지 — 둘은 다르며, 모호하면 거부가 안전).
3. 빈칸/미정의 쌍의 기본 정책을 **거부**로 못 박고, 거부 시 사람이 읽을 진단 메시지 포맷을 정한다.
4. 다축 동시 해소 규칙을 정의한다 — 두 노드는 6축을 동시에 맞춰야 하므로, 한 축이 거부면 전체 거부인지, 축별 어댑터를 합성하는지의 순서·우선순위를 명시.
5. 해소 엔진을 *순수 함수*로 구현한다: `resolve(portA_decl, portB_decl) → {ok, adapters[], reason}`. 런타임·UI 의존 0.
6. 격자 전체에 대한 망라적 속성 테스트(property test)를 작성한다 — "정의 안 된 칸이 단 하나도 없음"을 기계가 보증.

**리스크.**
- **가장 위험한 실패: 조용한 오접속.** push-stream을 single-pull에 꽂았는데 거부도 어댑터도 없이 첫 값만 먹고 나머지를 말없이 버리는 것 — "어디선 되고 어디선 안 되는" 병이 다른 가면으로 돌아온 것. → 빈칸 기본값을 거부로, 모든 "통과"는 명시적이어야 함.
- 어댑터 의미가 모호한 칸을 "대충 자동"으로 채우면 위와 같은 문제. → 의미가 둘 이상이면 거부하고 사람이 변환 노드를 명시하게.
- 격자가 손으로 채우기엔 클 수 있다(축·프리셋 늘면 제곱). → 4×6으로 닫고, 흔해지는 변환만 어댑터 규칙으로 승격(Phase 15·빈도 기반).

**의존.** 선행: Phase 1·2. 후행: 15(어댑터 자동삽입이 이 격자 소비), 16(컴파일러), 18(연결 시 UI가 호환/거부를 즉시 표시).

## Phase 4 — 와이어 포맷 & 메시지 엔벨로프

**목표.** 노드 간 실제로 흐르는 메시지의 전송(transport) 포맷을 정의한다. *보편 transport*와 *선택적 타입 레이어*를 분리해, 빡빡한 규칙이 transport에 닿지 않게 한다(동적↔정적 이음매 설계).

**스텝.**
1. 최소 공통 와이어 포맷을 정한다: `bytes` / `stream<bytes>`를 기저로, 위에 옵트인 타입드 페이로드(JSON Lines 등 합의된 envelope).
2. 엔벨로프 메타데이터를 정의한다: 메시지 ID, 출처 노드/포트, 타임스탬프, 시퀀스(스트림 순서), 그리고 신호 전용 메시지(페이로드 없음 = "트리거")의 표현.
3. in-process(같은 프로세스 내 타입드 노드 간, 직렬화 없이 참조 전달)와 out-of-process(직렬화 필수) 경로를 모두 정의하고 성능 특성을 명시.
4. 타입드 포트가 `any`/`bytes`를 만나는 *이음매(seam)*에서의 명시적 checked coercion 규약을 정한다 — 한 곳에서 시끄럽게 실패, 마찰이 그래프 전체로 번지지 않게.
5. 백워드 호환·버전 협상 규칙(노드 버전이 다를 때 엔벨로프를 어떻게 해석하는지)을 정의.
6. 와이어 포맷 인코더/디코더와, 라운드트립·악성 입력(잘린 스트림, 깨진 JSON) 테스트를 구현.

**리스크.**
- 직렬화 비용이 in-process 노드에서 병목이 될 수 있다. → 같은 프로세스 내 타입드 노드는 참조 전달로 우회, 직렬화는 프로세스 경계에서만.
- 스트림 순서/프레이밍을 와이어 레벨에서 안 잡으면 Phase 6(채널)·14(루프)에서 재현 불가 버그. → 시퀀스·프레이밍을 엔벨로프에 내장.

**의존.** 선행: Phase 1·2. 후행: 6(채널이 이 포맷 운반), 7(가드가 엔벨로프 검사), 9·10·11(호스트가 경계에서 직렬화), 13·16.

# Part II — 런타임 커널

> 노드들이 *살아서 돌고, 죽으면 되살아나고, 서로 메시지를 주고받는* 엔진. UI 없이 헤드리스로 완결되어야 한다(테스트 가능성). actor model + supervisor tree가 개념적 등뼈.

---

## Phase 5 — 노드 라이프사이클 상태 기계

**목표.** 단일 노드의 수명주기를 명시적 상태 기계로 구현한다. *함수가 아니라 완결된 실행 주기*라는 모델의 런타임 실체.

**스텝.**
1. 상태 집합을 정의한다: `created → initializing → running ⇄ idle/waiting → stopping → stopped`, 그리고 `failed`(+ 재시작 경로).
2. persistent 노드(계속 살아 메시지를 소비/생산)와 ephemeral 노드(트리거당 spawn→run→die)의 상태 전이 차이를 한 기계 안에서 모델링.
3. 각 상태 전이를 일으키는 이벤트(입력 도착, 트리거 발화, 정지 명령, 헬스 실패)를 정의하고 비결정적 전이를 배제.
4. 라이프사이클 콜백(Phase 2 SDK)을 상태 전이에 바인딩 — init/run/stop/health가 정확히 어느 전이에서 호출되는지 고정.
5. 노드 인스턴스에 안정적 ID(UUID, stop/start 생존)를 부여하고 상태를 외부에서 관찰 가능하게 노출(Phase 19·24가 소비).
6. 단일 노드를 무수히 spawn/stop/fail/restart하는 퍼징 테스트로 상태 기계의 불변(불가능 전이 없음)을 검증.

**리스크.**
- persistent/ephemeral를 별개 코드로 짜면 중복·불일치. → 하나의 상태 기계 + 플래그로 통합.
- `failed`에서 무한 재시작 루프. → 재시작 정책은 Phase 8(supervisor)이 소유하고, 상태 기계는 "재시작 요청"만 발생.

**의존.** 선행: Phase 2·4. 후행: 6·7·8(전부 이 상태 기계 위에), 14(루프의 재생성/재트리거가 이 전이를 사용), 24(상태 영속), 27(self-healing).

---

## Phase 6 — 엣지 채널 & 구조적 backpressure

**목표.** 엣지를 *bounded buffer를 가진 메시지 채널*로 구현한다. 불변식 7 — backpressure가 구조적으로 보장되는 지점.

**스텝.**
1. 채널을 bounded queue로 구현하고 용량(capacity)을 커넥션 속성으로 노출.
2. 버퍼 만수 시 정책을 정의·구현한다: 상류 블록(backpressure, 기본) / drop-oldest / drop-newest / 거부 — 커넥션별 선택 가능.
3. 단발(`single`)·스트림(`stream`)·요청응답(`request-response`)·브로드캐스트(`broadcast`) How-프리셋 각각의 채널 동작을 구현.
4. 다중 소비자(fan-out)와 다중 생산자(fan-in) 채널의 의미(복제 전달 vs 라운드로빈)를 명시·구현.
5. 채널 계측 훅(처리량/지연/버퍼 점유율)을 심어 Phase 19·25가 소비할 수 있게.
6. 빠른 생산자/느린 소비자, 폭주 fan-out 시나리오로 backpressure가 *상류를 실제로 막는지* 검증.

**리스크.**
- backpressure를 사용자 throttle 노드에 떠넘기면(수동) 폭주 케이스를 못 막는다. → 채널에 구조적으로 내장, throttle 노드는 보조.
- 무한 버퍼(unbounded)는 OOM·은밀한 지연. → 기본 capacity를 유한으로, unbounded는 명시 옵트인.
- 브로드캐스트 fan-out에서 한 느린 소비자가 전체를 막을 수 있다. → 소비자별 독립 버퍼 + 정책 분리.

**의존.** 선행: Phase 4·5. 후행: 13·14(루프 안전의 핵심), 16(컴파일러가 채널 용량 검증), 19(흐름 시각화), 25.

---

## Phase 7 — 경계 가드 (런타임 계약 강제)

**목표.** 노드가 선언한 포트 계약을 *런타임에 강제*한다. 불변식 2·3의 집행자 — "그래프는 거짓말하지 않는다"를 실제로 보증하는 단 하나의 메커니즘.

**스텝.**
1. 출력 경계 가드: 노드가 내보내는 모든 메시지가 그 포트의 선언(타입/스트림 모양)과 일치하는지 경계에서 검사, 위반 시 차단·보고.
2. 입력 경계 가드: 들어오는 메시지가 수신 포트 계약에 맞는지 검사(잘못 배선·악성 상류 방어).
3. 외부(약속 기반) 노드와 in-process(증명 가능) 노드에 대해 가드 강도를 분리 — 전자는 항상 런타임 검증, 후자는 컴파일 보증분만큼 생략 가능.
4. 가드 위반의 처리 정책을 정의: 노드 격리(fail로 전이)·연결 차단·진단 이벤트 — *위반의 영향이 노드 밖으로 새지 않음*(blast radius = 노드)을 보장.
5. 가드 자체의 성능 비용을 측정하고, hot path에서 검증을 표본화/캐시하는 전략(증명된 in-process 경로는 무가드)을 마련.
6. "잘못 선언한 노드"(선언과 실제 출력이 어긋남)를 일부러 만들어, 가드가 그래프와 실제 흐름의 괴리를 정확히 잡는지 검증 — 이것이 에이전트 생성 노드 신뢰의 토대(Phase 22 전제).

**리스크.**
- 가드가 없으면 에이전트가 잘못 선언한 노드 하나가 조용히 경계를 새서, 인간이 본 토폴로지 ≠ 실제 흐름이 된다(불변식 8·2 붕괴). → 모든 외부 포트는 가드 필수.
- 가드가 너무 비싸면 런타임이 느려진다. → in-process 증명 경로 무가드 + 경계만 검증.

**의존.** 선행: Phase 4·5. 후행: 16(컴파일러가 어느 가드를 생략 가능한지 판정), 22(에이전트 노드 봉쇄), 26(보안), 23.

---

## Phase 8 — Supervisor & 스케줄러

**목표.** 노드 인스턴스들의 수명주기를 *감독*하는 supervisor tree와, 트리거·시간 기반 실행을 관리하는 스케줄러를 구현한다(OTP式).

**스텝.**
1. supervisor 트리 구조를 정의 — 노드 그룹별 재시작 전략(one-for-one / one-for-all / rest-for-one)과 backoff·재시도 상한.
2. persistent 노드의 "항상 살아있게" 보장(크래시·재부팅 생존)과 ephemeral 노드의 "트리거당 spawn 후 회수"를 supervisor 정책으로 통합.
3. 스케줄러 코어 구현 — 시간/이벤트 트리거를 노드 발화로 변환, 정확한 발화 시각 보장.
4. 재시작 시 상태 복원 훅(Phase 24와 연동)과 "마지막 메시지 재생" 기본 동작을 정의.
5. supervisor 결정(재시작/포기/격리)을 관찰 가능 이벤트로 노출(Phase 19·25·27 소비).
6. 카오스 테스트 — 무작위로 노드를 죽이고 supervisor가 정책대로 복구하는지, 재시작 폭주가 backoff로 수렴하는지 검증.

**리스크.**
- 재시작 폭주(crash loop)가 자원을 태운다. → backoff + 상한 + 격리(반복 실패 시 포기하고 인간에게 보고).
- 스케줄러 시각 부정확(드리프트)으로 "새벽 1시" 류 트리거가 어긋난다. → 단조 시계 기반 + 드리프트 보정.

**의존.** 선행: Phase 5·6·7. 후행: **Part III 전체**(모든 호스트가 supervisor 아래 실행), 12(트리거 노드), 14, 27.

# Part III — 노드 호스트 (실행 기질)

> 실제로 다양한 종류의 노드를 *돌리는* 어댑터들. 전부 Phase 8 supervisor 아래에서 살고, Phase 2 SDK 계약·Phase 7 가드를 준수한다. 네 호스트는 의존 그래프상 병렬 개발 가능.

---

## Phase 9 — 프로세스 호스트 (언어 무관 실행 파일)

**목표.** 임의 언어로 작성된 외부 실행 파일을 노드로 실행하는 호스트. 불변식 2의 "약속 기반 노드"가 사는 곳.

**스텝.**
1. 프로세스 spawn/관리 구현 — stdin/stdout/stderr/exit-code를 Phase 4 와이어 포맷에 매핑.
2. 구조화 경계 어댑터 — 바이트 스트림 ↔ 합의된 envelope(JSON Lines) 변환을 경계에서 수행.
3. 라이프사이클 매핑 — 프로세스 시작/종료/크래시를 Phase 5 상태 기계 전이로 변환, exit-code를 완료/실패 신호로 해석.
4. 리소스 격리 기초 — 작업 디렉터리·환경변수·표준 스트림 외 접근 차단(Phase 26이 강화).
5. 외부 노드 가드 강제 연결 — 이 호스트의 모든 출력은 Phase 7 런타임 검증을 반드시 통과(선언≠증명이므로).
6. 3개 이상 언어의 샘플 노드로 동일 매니페스트 계약이 언어 무관하게 작동함을 회귀 테스트.

**리스크.**
- 외부 프로세스가 매니페스트 선언과 다르게 행동(약속 위반). → Phase 7 가드가 경계에서 차단(이 호스트는 가드 생략 불가).
- 좀비 프로세스·고아 자식. → supervisor가 프로세스 트리 전체를 회수.

**의존.** 선행: Phase 2·4·7·8. 후행: 22(에이전트가 만든 외부 노드 실행), 26, 28.

---

## Phase 10 — 터미널 호스트 (PTY 이중성)

**목표.** PTY 세션을 노드로 구현한다. 터미널-노드 *이중성* — 인간에겐 인터랙티브 패널, 워크플로엔 데이터 참여 노드.

**스텝.**
1. PTY 세션 관리(Rust pty crate) — spawn·리사이즈·종료, 세션 영속.
2. 출력 fan-out 구현 — PTY stdout이 (a) 터미널 렌더 뷰(Phase 20)와 (b) 다운스트림 노드, *양쪽*으로 흐르게.
3. 입력 fan-in 구현 — (a) 인간 키 입력과 (b) 워크플로가 주입하는 메시지가 *둘 다* stdin으로, 충돌 없이 다중화.
4. 경계 어댑터 — raw 바이트 스트림을 구조화 노드와 잇기 위한 라인/패턴 추출(이 변환도 노드, 불변식 1).
5. 세션 상태(작업 디렉터리, 살아있는 프로세스)를 라이프사이클 상태 기계에 노출.
6. 인간 입력과 프로그램 입력이 동시에 들어오는 경쟁 시나리오 테스트.

**리스크.**
- 인간/프로그램 입력 다중화 충돌(키 입력 중 주입). → 입력 큐 직렬화 + 모드 표시.
- PTY 바이트 스트림을 데이터 파이프로 직접 쓰면 취약(ANSI·재렌더). → 구조화가 필요하면 경계 어댑터로 명시 변환, 에이전트는 Phase 11의 구조화 모드 사용.

**의존.** 선행: Phase 2·4·5·8. 후행: 11(에이전트 호스트가 터미널 위에), 20(IDE 패널).

---

## Phase 11 — 에이전트 호스트 (구조화 스트리밍 + 프로비넌스)

**목표.** AI 에이전트(예: Claude Code류 CLI)를 노드로 구동한다. 불변식 11 — 스크레이프 금지, headless/JSON I/O.

**스텝.**
1. 에이전트를 headless/구조화 스트리밍 모드로 구동 — 구조화 입력(프롬프트 주입)·구조화 출력(응답/툴콜 이벤트) 채널 확보.
2. 이중 표면 구현 — 워크플로엔 구조화 메시지(Phase 4 envelope), 인간엔 동일 세션의 렌더링 터미널 뷰(Phase 10 위) 동시 제공.
3. 에이전트 세션을 ephemeral(작업당 spawn 후 회수, 불변식 10)로도, persistent(상시 대기)로도 운영 가능하게.
4. 프로비넌스 기록 — 세션 ID 발급, 노드 생성 작업이라면 생성된 노드에 ID 각인(불변식 9, Phase 23 연동).
5. 에이전트 특유 장애(컨텍스트 한계, rate-limit, 멈춤) 신호를 라이프사이클 이벤트로 노출(Phase 27 watchdog 소비).
6. 동일 명세로 여러 에이전트 세션을 병렬 spawn해 간섭 없음(stateless 독립성)을 검증.

**리스크.**
- 스크레이프 유혹(빠르니까). → headless 구조화 I/O로 못 박음(데이터플로 신뢰성·불변식 11).
- 에이전트 출력이 매니페스트 선언과 어긋남. → Phase 7 가드가 경계에서 검증.
- 헤드리스 모드가 인간 인터랙션 뷰와 상태 불일치. → 단일 세션을 두 표면이 공유하도록 설계.

**의존.** 선행: Phase 2·4·7·8·10. 후행: 22(에이전트 노드 생성 루프), 23(프로비넌스), 27(watchdog).

---

## Phase 12 — 트리거 & 내장 노드

**목표.** cron·타이머·메신저 등 흔한 진입/통합 노드를 1차 제공하고, *복잡 조건은 인라인 코드*로 붙이는 패턴을 확립한다("새벽 1시, 단 17일 제외" / "한도 차면 5시간 대기").

**스텝.**
1. cron/스케줄 트리거 노드 — Phase 8 스케줄러 위에서 발화, When=`on-trigger` 프리셋 구현.
2. 인라인 코드 게이트 노드 — 사용자가 흐름 중간에 박는 임의 조건(코드)으로 통과/차단·지연을 결정. `custom` 프리셋의 실체.
3. 흔한 선언적 중간 계층 노드 — 시간창 필터, 한도 대기(wait/cooldown), 디바운스 — *보이는* 게이트로 제공(코드에 숨기지 않음, 가시성 보존).
4. 메신저/이벤트 소스 노드(예: webhook 인입) — 외부 이벤트를 메시지로 변환하는 진입점.
5. 신호 전용 메시지(페이로드 없는 트리거)와 데이터 메시지를 동일 채널 의미로 통합 처리(불변식 4·How축).
6. "1시 발화·17일 제외", "rate-limit 시 5시간 대기" 등 대화에서 나온 구체 시나리오를 선언+인라인 조합으로 재현하는 e2e 테스트.

**리스크.**
- 흔한 조건(시간/대기)이 전부 인라인 코드로 숨으면 그래프가 프로그램을 덜 표현(가시성 침식). → 흔한 건 보이는 선언 노드, 코드는 진짜 임의 경우 탈출구로(가시성의 선, 불변식 8 정신).
- 인라인 코드 노드가 무제한 부작용. → Phase 7·26 격리 적용, 코드 노드도 일반 노드 경계 준수.

**의존.** 선행: Phase 5·6·8. 후행: 14(트리거가 루프 구동), 18·21(IDE에서 배치·코드 저작), 27.

# Part IV — 합성 & 컴파일

> 노드들을 *그래프로 엮고, 검증하고, 실행 가능한 형태로 컴파일*하는 계층. 불변식 1(폐포성)·5(빈칸 없음)·6(구조 strict/데이터 gradual)이 여기서 집행된다.

---

## Phase 13 — 그래프 모델 & 구조 컴파일

**목표.** 노드+엣지 그래프의 내부 표현과, *구조적* 정합성 컴파일(strict)을 구현한다. 불변식 6의 strict 절반.

**스텝.**
1. 그래프 데이터 모델 정의 — 노드 인스턴스, 포트, 엣지(=Phase 6 채널 바인딩), 레이어/적층, 사이클 허용.
2. 구조 검증기 구현 — 필수 입력 포트 미연결, 고립 노드, 리소스 미선언, 라이프사이클 모순을 strict-reject.
3. 사이클/루프 검출 — 루프의 *존재*를 식별하고, 종료/backpressure 안전장치 없는 루프를 구조 오류로 표시(Phase 14가 의미 부여).
4. 적층(layering) 의미 정의 — 그래프를 중첩/그룹화할 때의 경계와 포트 승격(subgraph as node) 규칙.
5. 구조 컴파일 결과를 명확한 진단(어느 노드/엣지/이유)으로 산출, UI가 인라인 표시 가능하게.
6. 대규모·악성 그래프(거대 사이클, 깊은 중첩)로 검증기 정확성·성능 테스트.

**리스크.**
- 구조 검증을 데이터타입 검증과 섞으면(불변식 6 위반) 이종 노드 도메인과 싸우게 된다. → 이 Phase는 구조만, 데이터는 Phase 16.
- subgraph-as-node 포트 승격이 모호하면 적층이 깨진다. → 승격 규칙을 Phase 1 프리셋과 정합하게 정의.

**의존.** 선행: Phase 2·6. 후행: 14·15·16(전부 이 모델 위에), 18(캔버스가 이 모델 편집), 19.

---

## Phase 14 — 루프 의미 & 반복 제어

**목표.** *라이프사이클을 가진 노드*가 포함된 루프의 의미를 확정·구현한다. stateless 데이터 루프와 본질적으로 다른 부분.

**스텝.**
1. 노드별 루프 행동 선언 — `regenerate`(매 반복 새 수명주기) vs `retrigger`(동일 영속 인스턴스, 다중 메시지). 둘 다 지원(Phase 5 전이 사용).
2. 반복 경계·종료 조건 정의 — 횟수/조건/외부 신호, 그리고 무한 루프 허용 여부와 안전장치.
3. 루프 내 backpressure 통합 — Phase 6 채널 용량으로 폭주 루프가 *구조적으로* 제어됨을 보장(수동 throttle 불필요한 기본 안전).
4. 루프 상태 캐리(iteration carry) — 반복 간 전달되는 상태의 명시적 모델(누가 보유하는가: 채널인가 영속 노드인가).
5. 빠른 루프에 시간/조건 제약을 *인라인 코드 노드*로 삽입하는 패턴 지원(Phase 12와 연동), 단 구조적 backpressure가 1차 방어임을 유지.
6. regenerate/retrigger × 빠른 루프 × backpressure 조합의 e2e 테스트(폭주가 실제로 수렴하는지).

**리스크.**
- regenerate와 retrigger를 사용자가 혼동하면 자원 누수/상태 오염. → 노드 속성으로 명시 + UI에서 시각 구분.
- "throttle 노드 박으면 됨"에만 의존하면 폭주 fan-out 미방어. → 구조적 backpressure가 기본, 인라인은 보조(불변식 7).

**의존.** 선행: Phase 5·6·13. 후행: 16(컴파일러가 루프 안전 검증), 19(루프 시각화), 27.

## Phase 15 — 변환 노드 & 어댑터 자동삽입

**목표.** 폐포성(불변식 1)을 구현한다 — 임피던스 불일치를 *같은 언어의 노드*로 흡수하고, 흔한 변환은 자동삽입으로 승격한다.

**스텝.**
1. 변환 노드를 일반 노드로 구현 — adapter·gate·throttle·collect·split 등이 입력/출력에서 서로 다른 프리셋 조합을 갖는 평범한 노드임을 코드로 실현.
2. Phase 3 해소 격자의 "어댑터삽입" 칸과 변환 노드를 매핑 — 연결 시 SDK가 적절한 어댑터를 자동으로 끼움.
3. 자동삽입과 수동삽입의 경계 정의 — 격자에 정의된 흔한 변환은 자동, 그 외는 사용자가 변환 노드를 명시.
4. **빈도 기반 승격 메커니즘** — 같은 수동 변환이 반복되면(예: 3회 이상) "격자 규칙 후보"로 표면화(흔한 건 규칙으로, 드문 건 노드로).
5. 변환 노드도 Phase 7 가드·Phase 5 라이프사이클을 준수함을 보장(특별 지위 없음).
6. 듬성한 기저 + 변환 합성으로 "표현 불가했던" 상호작용이 실제로 도달 가능함을 보이는 실효적 완전성 테스트.

**리스크.**
- 자동삽입이 의미 모호한 변환을 끼우면 조용한 오접속. → 격자에서 모호한 칸은 자동삽입 금지(거부 또는 수동).
- 흔한 변환을 매번 수동 재작성(폐포성은 "가능"이지 "편함"이 아님). → 빈도 승격으로 마찰을 0에 수렴.

**의존.** 선행: Phase 3·5·7·13. 후행: 16(컴파일 시 어댑터 확정), 18(연결 UX가 자동삽입 표시), 28.

## Phase 16 — 컴파일 파이프라인

**목표.** 그래프를 실행 가능한 런타임 구성으로 변환하는 전체 컴파일 파이프라인. 불변식 5·6의 종합 집행 — 구조 strict, 데이터 gradual, 외부는 런타임 가드.

**스텝.**
1. 파이프라인 단계 정의: 매니페스트 로드 → 구조 컴파일(Phase 13) → 해소/어댑터 확정(Phase 3·15) → 데이터타입 점진 검사 → 가드 배치 결정(Phase 7) → 실행 플랜 산출.
2. 데이터타입 검사를 soft/gradual로 구현 — 불일치는 hard-reject가 아니라 warn + 런타임 가드 삽입(파워유저 에토스 정합).
3. 가드 생략 최적화 — in-process 증명 경로는 무가드, 외부/약속 경로는 가드 필수(성능과 안전의 균형).
4. 컴파일 산출물 정의 — 어느 노드가 어느 supervisor 아래, 어느 채널 용량으로, 어떤 어댑터를 끼고 실행되는지의 완전한 실행 플랜.
5. 증분 컴파일 — 그래프 일부 변경 시 전체 재컴파일 없이 영향 부분만(IDE 반응성).
6. 컴파일 진단의 사람-가독성 — warning/error를 노드·엣지에 귀속해 Phase 18·19가 인라인 표시.

**리스크.**
- 데이터타입을 strict로 하면 이종 노드 도메인과 싸워 결국 거짓 엄격. → gradual + 런타임 가드(불변식 6).
- 가드 생략을 잘못 판정하면 안전 구멍 또는 과도한 비용. → 증명 가능성(in-process)을 보수적으로 판정.

**의존.** 선행: Phase 3·6·7·13·14·15. 후행: **17·18(IDE가 이 파이프라인 호출), 그리고 M3 최소 완성품의 직접 전제.**

# Part V — Tauri IDE (합성 표면)

> 이 도구의 정체성은 "코드 에디터"가 아니라 *구성 표면*이다 — 완결된 능력을 배치·배선하고, 흐름을 읽고·이해하고·신뢰하는 곳. 무게중심은 텍스트 편집이 아니라 *연결의 가독성*에 있다.

---

## Phase 17 — Tauri 셸 & 코어 IPC 브리지

**목표.** Tauri 앱 골격과, Rust 코어(Part II~IV 런타임/컴파일러) ↔ 웹뷰(UI) 사이의 IPC 브리지를 구축한다.

**스텝.**
1. Tauri 프로젝트 셋업 — 윈도우·메뉴·앱 수명주기, Rust 코어 크레이트 연결.
2. IPC 프로토콜 정의 — 명령(그래프 편집/컴파일/실행)과 이벤트(노드 상태/흐름/로그) 양방향 스트림.
3. 프로젝트 모델 — 그래프·노드·설정의 디스크 표현과 로드/세이브(Phase 24 영속과 정합).
4. 런타임 코어를 앱 내 헤드리스 서비스로 임베드 — Part II 엔진이 앱 프로세스(또는 사이드카)로 구동.
5. 백프레셔 있는 이벤트 스트림 — 런타임의 고빈도 이벤트(흐름/로그)가 UI를 압도하지 않게 throttle/집계.
6. 코어↔UI 왕복 지연·대용량 이벤트 부하 테스트.

**리스크.**
- 런타임 이벤트가 폭주해 웹뷰가 멎음. → 이벤트 스트림에 집계/표본화, 불변식 7 정신을 IPC에도.
- Rust 코어와 UI 상태 불일치(SSOT 모호). → 코어가 SSOT, UI는 투영(projection)으로 못 박음.

**의존.** 선행: **Part II~IV 핵심(특히 16).** 후행: 18·19·20·21(모든 UI가 이 브리지 위에), 24.

## Phase 18 — 그래프 캔버스 & 배선 UX

**목표.** 노드를 배치하고 엣지를 긋는 인간의 *구성* 경험. 불변식 8 — 인간이 토폴로지를 쥐는 표면.

**스텝.**
1. 캔버스 렌더링(React Flow류) — 노드/포트/엣지 시각화, 줌·팬·정렬, 적층/그룹 표현.
2. 연결 시 즉시 피드백 — Phase 3 해소 결과를 실시간 표시: 자동호환(붙음)·어댑터삽입(중간 변환 자동 표시)·거부(이유 툴팁). 불변식 5의 UX 발현.
3. 노드 팔레트 — 사용 가능한 노드(생성됨/패키지) 검색·배치, 매니페스트 기반 포트 미리보기.
4. 6축 선언의 시각 표현 — 포트가 어떤 프리셋인지 한눈에(색/아이콘), 인간이 "왜 이 둘이 붙는지/안 붙는지"를 보고 추론 가능하게.
5. 인라인 컴파일 진단 표시 — Phase 16의 warning/error를 해당 노드/엣지에 인라인.
6. 대규모 그래프(수백 노드)에서 배선·렌더 성능과 가독성 유지 테스트.

**리스크.**
- 6축을 다 노출하면 시각적 과부하. → 흔한 정보만 1차 표시, 상세는 드릴다운(인지 부하는 덧셈으로).
- 거부를 조용히 처리하면 "왜 안 붙지" 혼란. → 거부는 항상 이유와 함께 시끄럽게.

**의존.** 선행: Phase 13·15·16·17. 후행: 19·20·21(같은 캔버스 확장), 22(에이전트 배선 결과 표시).

## Phase 19 — 라이브 인트로스펙션 & 흐름 시각화

**목표.** *어느 노드가 살아있고, 뭘 기다리고, 어디서 막혔고, 루프가 도는지*를 그래프 위에서 본다. 이 도구가 이기거나 지는 축 — 그래프 가독성에 베팅.

**스텝.**
1. 노드 라이브 상태 오버레이 — Phase 5 상태(running/idle/waiting/failed)를 캔버스에 실시간 반영.
2. 엣지 흐름 시각화 — 메시지가 흐르는 엣지를 애니메이션·처리량 표시(Phase 6 계측 소비).
3. backpressure 가시화 — 버퍼 점유율·막힌 상류를 시각 신호로(폭주/병목을 한눈에).
4. 루프 시각화 — 반복 중인 루프, regenerate vs retrigger 구분 표시(Phase 14).
5. 노드 드릴다운 — 특정 노드의 최근 입출력·상태 이력·세션 링크(프로비넌스, Phase 23)로 진입.
6. 고빈도 상태 변화에서 시각화가 정확하고 UI가 버티는지 부하 테스트.

**리스크.**
- 시각화가 실제 런타임과 미세하게 어긋나면(이벤트 지연) 그래프가 거짓말하는 셈. → 이벤트 순서/시퀀스로 정합, 불일치 시 보수적 표시.
- 흐름 애니메이션이 성능을 잡아먹음. → 표본화·집계.

**의존.** 선행: Phase 5·6·14·17. 후행: 25(관측), 27(watchdog UI), 그리고 인간 검토 가능성(불변식 8)의 실현.

## Phase 20 — 임베디드 터미널/에이전트 패널

**목표.** 라이브 터미널·에이전트 세션을 IDE 안에서 *멀티플렉서처럼* 보고 조작한다. Phase 10·11의 인간 표면.

**스텝.**
1. 터미널 렌더러(xterm.js류) 임베드 — Phase 10 PTY 출력 fan-out의 인간 측 수신.
2. 인간 입력 경로 — 패널에서 키 입력이 Phase 10 fan-in으로(워크플로 주입과 다중화).
3. 멀티 세션 뷰 — 여러 터미널/에이전트 노드를 타일/탭으로 동시 관찰(멀티플렉서 경험).
4. 에이전트 세션 패널 — Phase 11 구조화 출력의 인간 가독 렌더 + 동일 세션 인터랙션, 워크플로 참여와 공존.
5. 세션-노드 연결 — 패널이 그래프의 어느 노드인지 양방향 하이라이트(캔버스↔패널).
6. 다수 라이브 세션 동시 렌더 시 성능·입력 정확성 테스트.

**리스크.**
- 패널 입력과 워크플로 주입 충돌(Phase 10 리스크의 UI 발현). → 입력 모드 표시·큐 직렬화.
- 다수 터미널 동시 렌더 성능. → 비활성 패널 렌더 보류.

**의존.** 선행: Phase 10·11·17·18. 후행: 22(에이전트 작업 관찰), 23(세션 되살리기 진입점).

## Phase 21 — 인라인 코드 노드 저작

**목표.** "복잡하면 그냥 코드" 탈출구의 저작 경험. 단, 로직이 코드로 *과도하게* 숨어 그래프 가시성을 침식하지 않도록 선을 의식.

**스텝.**
1. 코드 노드 에디터 — 임의 조건/변환을 인라인 코드로 작성(Phase 12 게이트·`custom` 프리셋 실체).
2. 코드 노드의 포트 선언 UX — 코드가 받는/내는 것을 5W1H로 선언(코드여도 경계는 명확, 불변식 3).
3. 코드 ↔ 그래프 가시성 가이드 — 흔한 패턴(시간/대기/디바운스)은 선언 노드 사용을 권유, 코드는 진짜 임의 경우로.
4. 코드 노드 격리 — 작성된 코드도 Phase 7·26 경계·샌드박스 준수(특별 지위 없음).
5. 에이전트 생성 코드 노드의 임포트 — Phase 22가 만든 코드 노드를 이 에디터로 검토/수정.
6. 잘못 짠 코드 노드(선언과 행동 불일치)가 Phase 7 가드에 잡히는지 검증.

**리스크.**
- 진짜 로직이 전부 코드 노드로 숨어 그래프가 "스크립트 둘레 장식"이 됨. → 흔한 건 선언 노드로 끌어내는 UX 넛지.
- 코드 노드 무제한 부작용. → 샌드박스·경계 가드 적용.

**의존.** 선행: Phase 7·12·17·18. 후행: 22·23(에이전트 코드 노드 검토/수정 경로).

# Part VI — 에이전트 저작 & 프로비넌스

> 불변식 8·9·10의 실현 — 에이전트는 stateless로 스폰되어 *노드만* 만들고, 만든 노드에 명세를 인라인하고 세션을 ID로 남긴다. 인간은 그 부품을 배선한다.

---

## Phase 22 — 에이전트 주도 노드 생성 루프

**목표.** 자연어 요청 → stateless 에이전트 세션 spawn → node-sdk 따라 노드 생성 → 등록까지의 자동 루프. 불변식 3·10의 작동.

**스텝.**
1. 노드 생성 요청 인터페이스 — 인간이 "이런 기능 노드가 필요"를 기술하면 에이전트 세션을 스폰(Phase 11).
2. SDK-가이드 생성 — 에이전트에게 node-sdk 계약·5W1H 프리셋·기본값을 컨텍스트로 제공(전체 코드베이스 아닌 *경계 계약만*, 불변식 3).
3. stateless 보장 — 세션은 과거 기억 없이 단일 노드 생성에 집중, 완료 후 회수(병렬 spawn 가능, 불변식 10).
4. 등록 파이프라인 — 생성된 노드의 매니페스트 검증(Phase 2 linter) → 격리 테스트 → 노드 팔레트 등록.
5. 봉쇄 검증 — 생성 노드를 Phase 7 가드 아래 격리 실행해 선언과 행동 일치를 확인(에이전트 신뢰의 런타임 근거).
6. 동일 요청을 다수 세션에 병렬 투입해 간섭 없음·등록 정합성 검증.

**리스크.**
- 에이전트가 매니페스트를 잘못 선언(약속≠행동). → Phase 7 가드 + 등록 전 격리 테스트가 필수 관문.
- 에이전트가 경계 밖을 가정(전체 이해 필요로 회귀). → SDK 컨텍스트를 경계 계약으로 한정, 그 이상 안 줌.

**의존.** 선행: Phase 2·7·11·21. 후행: 23(프로비넌스), 28(생성 노드 패키징).

---

## Phase 23 — 세션 프로비넌스 & 되살리기

**목표.** 불변식 9의 구현 — 명세는 노드에 인라인(재생성용), 세션 기록은 ID로 외부 참조(회상/"왜?"용). 서사와 산물의 분리.

**스텝.**
1. 명세 인라인 — 노드를 만든 5W1H 선언 + 요청 프롬프트를 노드 매니페스트에 박음(노드만 보고 재생성 가능).
2. 세션 기록 외부 저장 — 개발 세션의 전체 대화/추론을 외부 스토어에 보존, 노드엔 *주소(세션 ID)*만.
3. 되살리기 — "고치기"는 명세로 새 stateless 세션 재생성, "왜 이렇게?"는 ID로 외부 세션 기록 회상(두 경로 명확 분리).
4. 되살린 세션의 컨텍스트 복원 — 회상 시 원 세션 기록을, 재생성 시 명세를 각각 정확히 로드.
5. 프로비넌스 UI — Phase 19·20에서 노드 드릴다운 시 "고치기/왜?" 진입점 제공.
6. 명세로 재생성한 노드가 원본과 계약상 호환됨을(비결정성 하에서도 포트 계약 유지) 검증.

**리스크.**
- 세션 ID만 박고 기록을 안 보존하면 "왜?"가 영영 불가(불변식 9 미충족). → 회상을 원하면 기록 외부 보존 필수.
- 재생성=회상으로 혼동. 재생성은 다른 노드가 나올 수 있음(서사 복원 아님). → 두 경로를 UI·의미상 분리.

**의존.** 선행: Phase 2·11·22. 후행: 24(기록 영속), 25.

# Part VII — 하드닝 & 운영

> "데모는 되는데 못 믿는 것"과 "새벽 1시에 자러 가도 되는 것"의 차이가 여기서 갈린다. 각 Phase는 선행이 다르며 상당 부분 병렬 가능.

---

## Phase 24 — 영속 & 크래시 복구

**목표.** 세션·그래프·런타임 상태가 재시작·크래시를 생존한다. supervisor의 복원 훅(Phase 8)에 실체를 부여.

**스텝.**
1. 그래프/프로젝트 영속 — 디스크 저장 포맷과 원자적 저장/로드(Phase 17 프로젝트 모델 확장).
2. 런타임 상태 스냅샷 — persistent 노드의 상태·채널 버퍼·루프 캐리를 주기적/체크포인트 저장.
3. 재시작 복원 — 크래시 후 supervisor가 마지막 일관 상태로 노드를 되살리고 "마지막 메시지 재생".
4. 세션 기록 영속 — Phase 23 외부 세션 스토어의 내구 저장.
5. 부분 실패 복구 — 일부 노드만 죽었을 때 전체 재시작 없이 격리 복원.
6. 무작위 크래시 주입 후 상태 정합·중복 처리 없음(at-least/at-most-once 의미 명시)을 검증.

**리스크.**
- 재생이 부작용을 중복 실행(비멱등 노드). → 멱등성 경계/시퀀스로 중복 억제, 의미를 명시.
- 스냅샷이 비일관(레이스). → 체크포인트 일관성 프로토콜.

**의존.** 선행: Phase 5·8·17·23. 후행: 27(self-healing이 복구 위에).

## Phase 25 — 관측 & 비용 추적

**목표.** 로그·메트릭·트레이스를 그래프 전반에 걸쳐 수집하고, 에이전트 노드의 토큰/비용을 추적한다.

**스텝.**
1. 구조화 로깅 — 노드·엣지·supervisor 이벤트를 귀속 가능한 형태로(Phase 5·6·8 계측 소비).
2. 그래프 트레이싱 — 한 트리거가 그래프를 따라 흐른 경로·지연을 추적(분산 트레이스 유사).
3. 토큰/비용 추적 — 에이전트 노드(Phase 11)의 세션별 사용량·비용 집계.
4. 메트릭 대시보드 — 처리량·지연·버퍼 점유·실패율을 Phase 19와 연동 표시.
5. 로그/트레이스 보존·회전 정책 — 디스크 압박 방지(불변식 7 정신).
6. 고부하에서 관측 오버헤드가 런타임을 왜곡하지 않는지 측정.

**리스크.**
- 관측 자체가 hot path를 느리게. → 비동기·표본화.
- 트레이스가 루프에서 폭발. → 루프 트레이스 집계/샘플.

**의존.** 선행: Phase 5·6·8·11·19. 후행: 27·29.

## Phase 26 — 보안·샌드박싱·blast-radius 강제

**목표.** 불변식 3을 OS 레벨까지 강제 — 노드의 영향이 선언된 경계 밖으로 새지 않게.

**스텝.**
1. 프로세스 샌드박싱 — 외부/코드/에이전트 노드의 파일시스템·네트워크·시스템콜 접근을 매니페스트 리소스 선언으로 제한.
2. 시크릿 관리 — 노드에 필요한 자격증명의 안전한 주입·격리(로그·프로비넌스에 누출 금지).
3. 네트워크 노출 제어 — 로컬 우선, 외부 노출은 명시 옵트인 + 기본 차단(원격 접근은 안전 경로로만).
4. 코드/에이전트 노드 격리 강화 — Phase 21·22 산출물이 임의 부작용을 못 내게 샌드박스 안에서 실행.
5. 권한 경계 감사 — 어떤 노드가 어떤 리소스에 접근하는지 가시화·검토(인간 권한, 불변식 8).
6. 탈출 시도(샌드박스 브레이크, 시크릿 추출) 레드팀 테스트.

**리스크.**
- 매니페스트 리소스 선언과 실제 접근 불일치(약속≠행동). → 샌드박스가 *선언분만* 허용(가드+OS 강제).
- 시크릿이 세션 기록/로그로 새어 프로비넌스에 잔존. → 스크러빙·격리.

**의존.** 선행: Phase 7·9·11·21. 후행: 28·29.

## Phase 27 — Self-Healing Watchdog

**목표.** 무인 장시간 운전을 가능케 하는 자가치유 계층 — 에이전트 한계, rate-limit, 멈춤, 크래시를 자동 복구. (AMUX류 watchdog 아이디어를 구조화 신호 위에서 재구현.)

**스텝.**
1. 헬스 모니터 — Phase 5 상태·Phase 8 supervisor·Phase 11 에이전트 신호를 종합 감시.
2. rate-limit 처리 — 에이전트 한도 도달 신호 감지 → 대기 → 리셋 후 자동 재개("한도 차면 5시간 대기"의 시스템화). 구조화 신호 기반(스크레이프 아님, 불변식 11).
3. 컨텍스트/멈춤 복구 — 에이전트 컨텍스트 한계·멈춤 시 정책적 압축/재시작/재개.
4. fleet 인지 — 동일 계정 다수 세션이 동시에 막힐 때 일괄 처리·일괄 재개.
5. 복구 정책 설정 — 자동 재개 상한/모드(off/capped/unlimited 유사)와 인간 개입 감지(수동 개입 시 자동 재개 보류).
6. 장시간(수일) 무인 운전 시뮬레이션으로 자가치유 안정성·자원 수렴 검증.

**리스크.**
- 자동 재개가 폭주/비용 폭발. → 상한·backoff·예산.
- 인간이 개입한 세션을 watchdog이 덮어씀. → 개입 감지 후 자동 동작 보류.

**의존.** 선행: Phase 5·8·11·24·25. 후행: 29(무인 운전이 릴리스 품질 기준).

## Phase 28 — 노드 패키징·버저닝·재사용

**목표.** "한 번 만들면 어디든 붙는다"의 운영적 실현 — 노드를 버전·패키지로 묶어 프로젝트·기계 간 재사용. (공유 플랫폼은 비목표지만, 미래의 자신을 위한 재사용성은 목표.)

**스텝.**
1. 노드 패키지 포맷 — 매니페스트 + 구현 + 인라인 명세(Phase 23) + 의존을 한 단위로.
2. 버저닝·호환 — Phase 1·2의 하위호환 규칙을 패키지 레벨로(프리셋/포트 변경의 semver 의미).
3. 로컬 레지스트리 — 만든 노드를 기계 단위로 보관·검색·재사용(팔레트 연동, Phase 18).
4. 의존 해소 — 노드가 다른 노드/리소스에 의존할 때의 해소·버전 잠금.
5. 재현성 — 동일 패키지가 다른 환경에서 동일 계약으로 작동함을 보장(언어 무관 노드 포함, Phase 9).
6. 버전 충돌·누락 의존 시나리오의 복구 가능성 테스트.

**리스크.**
- 노드가 "어디선 되고 어디선 안 됨"(환경 의존 누설). → 리소스/의존을 매니페스트에 명시 + 샌드박스로 환경 격리.
- 버전 변경이 기존 그래프를 조용히 깸. → semver + 컴파일 시 비호환 경고(Phase 16).

**의존.** 선행: Phase 2·9·15·22·26. 후행: 29.

## Phase 29 — 하드닝·eval 하니스·릴리스

**목표.** 전 시스템의 통합 검증, 회귀 방지 eval, 그리고 배포. "믿고 무인으로 돌릴 수 있는가"의 최종 관문.

**스텝.**
1. e2e 통합 시나리오 — 대화에서 나온 실제 케이스("1시·17일 제외 cron → 에이전트 노드 → 변환 → 출력", "rate-limit 대기 루프")를 전 스택 관통 테스트.
2. eval 하니스 — 노드 생성(Phase 22) 품질, 해소 격자 정확성(Phase 3), 가드 강제(Phase 7)를 회귀 측정.
3. 성능·부하 — 대규모 그래프 + 다수 라이브 세션 + 장시간 무인의 종합 부하.
4. 불변식 감사 — 11개 불변식이 통합 환경에서 실제로 유지되는지 체계적 점검(특히 "그래프는 거짓말 안 한다").
5. 패키징·배포 — Tauri 빌드, 자동 업데이트, 플랫폼별 배포·서명.
6. 도그푸딩 — 이 IDE로 이 IDE의 일부 워크플로(예: 빌드/eval 파이프라인)를 운영해 실사용 검증.

**리스크.**
- 통합에서야 드러나는 Phase 간 의미 불일치(특히 라이프사이클×루프×backpressure×가드 교차). → 불변식 감사를 통합 단계의 1급 관문으로.
- "데모는 되는데 무인 신뢰 불가". → Phase 27 장시간 운전 + 불변식 감사 통과를 릴리스 기준으로.

**의존.** 선행: **사실상 전체.** 후행: 없음(릴리스). 이후는 운영·반복.

## 3. 마일스톤 (통합 체크포인트)

Phase를 가로지르는 "동작하는 무언가"의 정의. 각 마일스톤은 그 자체로 검증 가능한 가치를 낸다.

- **M0 — 프로토콜 동결 (Phase 1~4).** 5W1H 명세·매니페스트·해소 격자·와이어 포맷이 문서+순수 구현으로 확정. 코드는 안 돌지만 *모든 후속의 기준*이 고정됨. 이걸 못 넘기면 뒤가 전부 흔들린다.
- **M1 — 헤드리스 커널 (Phase 5~8).** UI 없이 노드를 spawn·연결·감독·복구. CLI로 "두 노드를 채널로 잇고 메시지 흘리기"가 된다.
- **M2 — 살아있는 노드들 (Phase 9~12).** 외부 실행 파일·터미널·에이전트·cron이 노드로 돈다. 헤드리스로 "cron이 에이전트 노드를 트리거"가 된다.
- **M3 — 컴파일되는 그래프 (Phase 13~16).** 그래프가 검증·컴파일·실행된다. **최소 완성품**: 텍스트 그래프 정의를 컴파일해 헤드리스로 돌리기. (임계 경로의 끝.)
- **M4 — 보이는 IDE (Phase 17~21).** Tauri에서 노드를 배치·배선하고 흐름을 보며 터미널/에이전트를 조작. *인간이 구성하는* 경험 완성.
- **M5 — 에이전트가 부품을 만든다 (Phase 22~23).** stateless 에이전트가 노드를 생성·등록하고, 인간이 배선. 프로비넌스로 되살리기. *생성/권한 분리*가 작동.
- **M6 — 무인 신뢰 (Phase 24~29).** 영속·관측·보안·자가치유·패키징·통합. "새벽 1시에 자러 가도 되는" 상태. 릴리스.

## 4. 결정됐으나 계속 주시할 항목

대화에서 결론지었지만, 구현 중 재확인이 필요한 지점들.

1. **서사 보존 = 주소 + 명세 둘 다 (Phase 23).** 명세는 노드 인라인(재생성), 세션 기록은 ID 외부 참조(회상). "고치기"와 "왜?"가 분리됨을 끝까지 유지.
2. **데이터타입은 끝까지 gradual (Phase 16).** strict로 끌어올리고 싶은 유혹이 반복될 것. 외부 노드는 *약속*이라는 사실(불변식 2)이 이를 금한다.
3. **backpressure는 구조적 1차 / 인라인 보조 (Phase 6·14).** "throttle 노드 박으면 됨"으로 후퇴하지 않기.
4. **프리셋은 4로 닫고 격자는 작게 (Phase 1·3·15).** 표현 부족분은 `custom`+변환 노드 + 빈도 승격으로. 축/프리셋을 늘려 격자를 키우려는 압력에 저항.
5. **에이전트 구동은 구조화 스트리밍 (Phase 11).** 스크레이프가 빠르다는 유혹에 후퇴하지 않기 — 데이터플로 신뢰성의 근간.
6. **킬러 경로 하나.** 최대 일반성(모든 노드·조합·루프)을 추구하되, 사람이 들어올 *명백히 제일 잘하는* 한 워크플로를 함께 가져가기(예: 무인 에이전트 오케스트레이션). 일반 substrate ≠ 무목적.
