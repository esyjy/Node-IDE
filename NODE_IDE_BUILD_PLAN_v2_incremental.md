# 노드-엣지 워크플로 IDE — 증분 개발 플랜 (v2, 버전 기반 재구성)

> 이 문서는 이전 "29 phase 수평 레이어" 플랜을 대체한다. 핵심 차이: **모든 단계는 실제로 실행되는 완결된 Tauri GUI 앱**이며, 단계는 *수직 슬라이스*로 살을 붙여나간다. v1이 안 돌고 v17부터 도는 일은 없다. 각 버전은 인앱 업데이트로 배포되고, 너는 매 버전을 실행해 직접 도그푸딩한다.

---

## 0. 이 문서의 읽는 법

이 플랜은 "구현을 24개로 쪼갠 것"이 아니라 **24개의 출시 가능한 버전**이다. 각 버전은:

- 실제로 빌드되어 실행되는 완결 앱이다(walking skeleton → 점점 두꺼워짐).
- v(n)은 v(n-1)이 하던 **모든 것** + 증분을 한다. **어떤 버전도 퇴행하지 않는다.**
- 인앱 "업데이트" 버튼으로 전달되며, 너는 업데이트→실행→사용→문제 발견→수정의 루프를 돈다.

각 버전 항목의 블록:

- **실행하면 (완결 앱):** 이 버전을 실행했을 때 end-to-end로 가능한 것. "완결됨"의 구체적 정의.
- **더해지는 것:** v(n-1) 대비 증분(누적·비퇴행).
- **네가 직접 (도그푸딩):** 업데이트해서 실제로 해보는 것 + 거기서 드러날 법한 것.
- **리스크:** 깨지거나 잘못 결정되기 쉬운 지점.
- **의존:** 선행 버전 + 비자명한 교차 의존.

버전은 7개 **era**로 묶인다. era는 권장 순서이며, 각 era의 끝은 "더 유능하지만 여전히 완결된 앱"이다.

---

## 1. 불변식

이전 문서의 **제품 불변식 11개**(일급 시민은 노드, 그래프는 거짓말 안 함, 경계=blast radius=knowledge radius, 프리셋 격자 빈칸 없음[기본=거부], 구조 strict/데이터 gradual, backpressure 구조적, 생성/권한 분리, 서사/산물 분리, 에이전트 stateless 스폰, 구동은 구조화 스트리밍)는 **그대로 유효하다.** 전송 구조와 무관하기 때문이다.

여기에 이번 재구성의 **배포 불변식 6개**를 추가한다. 어떤 단계 결정도 이 6개와 충돌하면 그 결정이 틀린 것이다.

- **D1 — 모든 버전은 실행되는 완결 앱이다.** 헤드리스 "층"을 쌓다 마지막에 합치는 방식 금지. v(n)은 v(n-1)의 전부 + 증분, 퇴행 없음.
- **D2 — 수직 슬라이스, 수평 레이어 아님.** 각 버전은 모든 계층(UI↔런타임↔노드)을 관통하는 *가장 얇은 end-to-end 슬라이스*다. 프로토콜/커널/컴파일러를 따로 완성하지 않는다.
- **D3 — 프로토콜은 추가로만 자란다.** 배포된 프리셋/축의 의미는 절대 변경 금지. 새 축·프리셋은 *추가만* 된다. 그래서 과거 그래프는 업데이트 후에도 계속 작동한다. ("동결"이 아니라 "배포 시 동결 + 추가 성장".)
- **D4 — 업데이트가 배포 수단이다.** 모든 버전은 인앱 업데이트로 전달된다. 업데이터는 v1의 일부이며, 지속적으로 도그푸딩·하드닝된다.
- **D5 — 업데이트는 마이그레이션을 동반한다.** 노드 포맷·그래프 스키마·영속 상태가 버전 간 바뀌면 업데이트가 기존 데이터를 자동 이주시킨다. "업데이트 눌렀더니 내 그래프가 깨짐"은 금지.
- **D6 — GUI가 일급, CLI는 보조.** 처음부터 Tauri GUI 앱이다. CLI는 같은 앱이 노출하는 보조 인터페이스(스크립팅/자동화/헤드리스 회귀 테스트)일 뿐, 초기 대체물이 아니다.

---

## 2. 업데이트 & 마이그레이션 척추 (cross-cutting)

네가 가장 중요하다고 한 부분. 한 버전의 기능이 아니라 *전 버전을 관통하는 메커니즘*이므로 따로 둔다.

- **업데이터는 v1에 들어간다.** 업데이트할 내용이 거의 없어도 메커니즘(업데이트 확인·다운로드·서명 검증·적용·재시작)은 v1부터 존재한다. v2가 이 메커니즘의 첫 실전 검증이다.
- **서명·배포 파이프라인을 v1에서 핀(pin)한다.** 서명된 릴리스 아티팩트, 업데이트 엔드포인트/채널, 롤백 경로. (Tauri는 updater 역량을 제공한다 — 구체 플러그인/API 버전은 v1 착수 시 확정. 이 문서는 의미만 명시.)
- **마이그레이션 레지스트리.** 영속 데이터(그래프·노드 패키지·런타임 스냅샷)에 스키마 버전을 박고, 버전 간 마이그레이터를 등록·체이닝한다. 업데이트 적용 시 자동 실행.
- **업데이트 안전망.** 마이그레이션 실패 시 자동 백업 복원 + 롤백. 업데이트 전 데이터 스냅샷은 의무.
- **업데이터 자체의 도그푸딩.** 각 버전이 업데이트로 배포되는 과정에서 드러나는 업데이터/마이그레이션 버그는 *다음 버전의 1순위 수정 항목*으로 취급한다(별도 phase가 아니라 상시 관심사).

이 척추가 D3(추가로만 성장)·D5(마이그레이션)와 맞물려야 "업데이트 눌러서 다음 버전 받기 + 내 작업 보존"이 실제로 성립한다.

## 3. 버전 로드맵 개요 (누적)

```
Era A · 골격과 업데이트 루프      v1  v2  v3  v4    ← 항상 실행되는 최소 완결 앱 + 자기 업데이트
Era B · 진짜 이종 노드            v5  v6  v7        ← 외부 실행파일/터미널/에이전트가 노드로
Era C · 흐름 제어                 v8  v9  v10 v11   ← 트리거/인라인코드/backpressure/루프
Era D · 완전한 프로토콜과 합성    v12 v13 v14 v15   ← 6축 완성/변환·어댑터/컴파일/가드
Era E · 에이전트 저작            v16 v17           ← 에이전트가 노드 생성, 인간이 배선, 프로비넌스
Era F · 무인 운전                v18 v19 v20 v21   ← supervisor·자가치유/영속/관측/보안
Era G · 재사용과 성숙            v22 v23 v24       ← 패키징·재사용/멀티플렉서 성숙/통합·릴리스

누적 규칙: 오른쪽 버전은 왼쪽 전부를 포함한다. 화살표는 시간이지 의존-조립이 아니다.
업데이트 척추(§2)는 v1부터 v24까지 모든 버전을 관통한다.
```

가장 중요한 단일 버전은 **v1**이다 — "실행되고 자기 업데이트되는 최소 앱"이 서면, 이후 모든 버전이 *그 위에 업데이트로 흘러들어올* 통로가 열린다. v1이 임계 경로의 시작이자 도그푸딩 루프의 점화다.

# Era A — 골격과 업데이트 루프

> 목표: "언제나 실행되고, 스스로 업데이트하며, 노드를 잇는 가장 얇은 완결 앱". 여기서부터 너는 매 버전을 실제로 실행해 만진다.

---

## v1 — Walking Skeleton + 자기 업데이트

**실행하면 (완결 앱).** Tauri 앱이 창을 띄운다. 캔버스에 *하나의 내장 노드*(예: "상수 출력" 또는 "에코")를 놓고 Run을 누르면 실행되어 결과가 패널에 뜬다. 그리고 메뉴의 "업데이트 확인"이 작동한다(서명 검증·적용·재시작). 작지만 완전히 실행되는 앱이다.

**더해지는 것 (= 전부, 토대).**
- Tauri 셸 + Rust 코어 ↔ 웹뷰 IPC 브리지.
- 최소 노드 모델 + 단일 노드를 실행하는 최소 런타임(라이프사이클 씨앗: created→running→done).
- 캔버스(React Flow류) 골격: 노드 1개 배치·실행·결과 표시.
- **인앱 업데이터**(§2) + 서명/배포 파이프라인 핀 + 마이그레이션 레지스트리 골격(빈 상태라도 존재).
- 같은 동작의 CLI 진입점(보조, D6).

**네가 직접 (도그푸딩).** 빌드해서 실행 → 노드 하나 돌려보기. 그다음 v2가 나오면 "업데이트 확인"을 눌러 v2로 갱신. **이 첫 업데이트에서 거의 확실히 문제가 난다**(서명/엔드포인트/재시작/권한) — 그 수정이 v2의 일부.

**리스크.**
- 업데이터를 나중으로 미루고 싶은 유혹 → 미루면 도그푸딩 루프 자체가 안 돈다. v1 필수(D4).
- 마이그레이션 레지스트리를 안 깔면 v3쯤 포맷이 바뀔 때 기존 그래프가 깨진다 → 빈 레지스트리라도 v1에 심는다(D5).
- 코어↔UI 상태 진실 원천 모호 → 코어가 SSOT, UI는 투영으로 못 박음.

**의존.** 선행: 없음(루트). 후행: **전부**(모든 버전이 v1의 셸·IPC·업데이터·마이그레이션 위에 흘러든다).

---

## v2 — 엣지와 메시지 흐름

**실행하면 (완결 앱).** 노드 *두 개*를 놓고 엣지로 이어, A의 출력이 B로 흘러 결과를 본다. 이제 진짜 "노드-엣지" 앱이다. (그리고 v1→v2 업데이트가 처음으로 실전 검증된다.)

**더해지는 것.**
- 엣지 = 메시지 채널(최소): 단발 메시지 A→B.
- 메시지 엔벨로프 최소형(출처·시퀀스)·와이어 포맷 v1.
- 캔버스 배선 UX(드래그로 포트 연결).
- 업데이터 1차 하드닝(v1 도그푸딩에서 나온 버그 수정).

**네가 직접.** 두 노드를 이어 메시지를 흘려보기. 업데이트 경로가 v1보다 매끄러운지 확인.

**리스크.**
- 채널 프레이밍/순서를 와이어에서 안 잡으면 뒤(스트림·루프)에서 재현 불가 버그 → 엔벨로프에 시퀀스 내장.
- 엣지 의미를 너무 일찍 과하게 정의 → v2는 "단발 메시지" 하나로 시작, 나머지는 v10·v12에서 추가(D3).

**의존.** 선행: v1. 후행: 거의 전부(흐름의 기초).

---

## v3 — 최소 5W1H 선언 + 연결 피드백

**실행하면 (완결 앱).** 노드 포트가 **2축(What·How)**으로 자신을 선언하고, 두 포트를 이으려 할 때 앱이 즉시 *호환/거부*를 보여준다. 호환이면 붙고, 거부면 이유가 뜬다. 부분적이지만 선언형 계약이 처음으로 작동한다.

**더해지는 것.**
- What(무엇이 흐르나)·How(single/stream/req-resp/broadcast) 2축 + `custom`. 매니페스트 v1.
- 이 2축에 대한 해소 격자(빈칸=거부) + 연결 시 즉시 피드백.
- node-sdk 최소 계약: `any` I/O + no-op 라이프사이클 기본 상속(한 줄 노드).
- **프로토콜 버저닝 + addition-only 규칙 활성화**(D3) — 이후 축은 추가만.

**네가 직접.** 일부러 안 맞는 포트를 이어 거부 메시지가 *읽고 고칠 수 있게* 나오는지 확인. (이 "거부가 사람에게 읽히는가"가 평생 가는 품질 축.)

**리스크.**
- 2축이 사실 직교 아님(How=stream이 다른 걸 강제) → 손으로 교차 검증.
- 조용한 오접속(거부도 어댑터도 없이 첫 값만 먹고 버림) → 빈칸 기본값 거부, 모든 통과는 명시(불변식 5).

**의존.** 선행: v2. 후행: v12(나머지 4축 추가), v13·v14·v15.

---

## v4 — 라이프사이클 가시화 + persistent/ephemeral

**실행하면 (완결 앱).** 노드의 상태(idle/running/waiting/done/failed)가 캔버스에 *실시간*으로 보인다. 상시 가동(persistent) 노드와 일시(ephemeral) 노드를 둘 다 놓고 차이를 눈으로 본다.

**더해지는 것.**
- 노드 라이프사이클 상태 기계(persistent ⇄ ephemeral, 플래그 통합).
- 캔버스 라이브 상태 오버레이(흐름 시각화의 씨앗).
- 안정적 노드 인스턴스 ID(stop/start 생존).
- 상태 이벤트 IPC 스트림(집계/표본화로 UI 보호).

**네가 직접.** persistent 노드를 띄워두고, ephemeral 노드를 반복 실행하며 상태 전이를 관찰. UI가 고빈도 상태 변화에 버티는지.

**리스크.**
- persistent/ephemeral를 별개 코드로 → 하나의 상태 기계 + 플래그로.
- 상태 이벤트 폭주로 웹뷰 멎음 → IPC에 집계/throttle.

**의존.** 선행: v1·v2. 후행: v8·v10·v11·v18(전부 라이프사이클 위에).

# Era B — 진짜 이종 노드

> 목표: 장난감 내장 노드를 넘어, *실제* 외부 실행파일·터미널·에이전트를 노드로 돌린다. 여기부터 앱이 "쓸모"를 갖기 시작한다.

---

## v5 — 프로세스 노드 (언어 무관 실행파일)

**실행하면 (완결 앱).** *네 임의 스크립트*(파이썬/노드/러스트 바이너리 등)를 노드로 감싸 캔버스에서 실행하고, stdin으로 넣고 stdout을 받아 다음 노드로 흘린다. 첫 "진짜" 노드.

**더해지는 것.**
- 프로세스 호스트: spawn/관리, stdin/stdout/stderr/exit-code ↔ 와이어 매핑.
- 경계 어댑터: 바이트 ↔ JSON Lines envelope.
- exit-code를 완료/실패 신호로 → 라이프사이클 전이.
- 외부 노드 = "약속" → 경계 검증 최소형(가드의 씨앗, v15에서 강화).

**네가 직접.** 평소 쓰던 스크립트 하나를 노드로 만들어 실제 흘려보기. 3개 언어로 같은 계약이 먹히는지.

**리스크.**
- 외부 프로세스가 선언과 다르게 행동(약속≠행동) → 경계 검증(v15가 본격화).
- 좀비/고아 자식 → 프로세스 트리 회수.

**의존.** 선행: v1·v2·v4. 후행: v16(에이전트가 만든 외부 노드 실행), v21·v22.

## v6 — 터미널 노드 + 임베디드 터미널 패널

**실행하면 (완결 앱).** 살아있는 터미널(PTY)을 노드로 띄우고, 앱 안의 터미널 패널에서 *직접 타이핑*하면서 동시에 그 출력이 다운스트림 노드로도 흐른다. 멀티플렉서 경험의 씨앗.

**더해지는 것.**
- PTY 호스트(Rust pty): spawn·리사이즈·종료·세션 영속.
- 출력 fan-out(패널 + 다운스트림), 입력 fan-in(인간 키 + 워크플로 주입) 최소형.
- 임베디드 터미널 렌더러(xterm.js류).
- raw 바이트 ↔ 구조화 노드 경계 어댑터(라인/패턴 추출, 이것도 노드).

**네가 직접.** 패널에서 명령을 치면서, 같은 세션의 출력이 노드 흐름으로 빠지는 걸 확인. 인간 입력과 주입이 충돌 안 하는지.

**리스크.**
- 인간/프로그램 입력 다중화 충돌 → 입력 큐 직렬화 + 모드 표시.
- 바이트 스트림을 데이터 파이프로 직접 쓰면 취약 → 구조화 필요 시 경계 어댑터로 명시.

**의존.** 선행: v1·v4·v5. 후행: v7(에이전트가 터미널 위에), v23(멀티플렉서 성숙).

## v7 — 에이전트 노드 (구조화 스트리밍)

**실행하면 (완결 앱).** AI 에이전트(예: Claude Code류 CLI)를 노드로 띄워, 패널에서 인간이 보고 조작하는 *동시에* 워크플로가 구조화 메시지로 프롬프트를 주입하고 응답을 받는다. 불변식 11: 스크레이프 아님, headless/JSON I/O.

**더해지는 것.**
- 에이전트 호스트: headless/구조화 스트리밍 구동(구조화 입력/출력 채널).
- 이중 표면: 워크플로엔 구조화 메시지, 인간엔 동일 세션 렌더 뷰(v6 위).
- ephemeral(작업당 spawn 후 회수, 불변식 10) / persistent 운영.
- 에이전트 장애 신호(컨텍스트 한계·rate-limit·멈춤)를 라이프사이클 이벤트로 노출(v18 watchdog 씨앗).

**네가 직접.** 에이전트 노드에 작업을 주입하고 결과를 흐름으로 받기. 같은 명세로 여러 세션 병렬 spawn해 간섭 없음 확인.

**리스크.**
- 스크레이프가 빠르다는 유혹 → headless 구조화로 못 박음(데이터플로 신뢰성).
- 에이전트 출력이 선언과 어긋남 → 경계 검증(v15).

**의존.** 선행: v6. 후행: v16(에이전트 노드 생성), v18.

# Era C — 흐름 제어

> 목표: 수동 Run을 넘어 *시간·이벤트·조건·반복*으로 그래프가 스스로 흐르게. "새벽 1시" 류 시나리오가 여기서 처음 실현.

---

## v8 — 트리거 & 내장 노드 (cron/타이머/이벤트)

**실행하면 (완결 앱).** cron/타이머 노드를 놓으면 시각에 맞춰 다운스트림이 자동 발화한다. webhook 같은 외부 이벤트도 진입점으로. 이제 그래프가 사람 없이 트리거된다.

**더해지는 것.**
- 스케줄러 코어(단조 시계 기반, 드리프트 보정) + cron/타이머 트리거 노드.
- 이벤트 소스 노드(webhook 인입 등).
- 신호 전용 메시지(페이로드 없는 "트리거")와 데이터 메시지 통합 처리.
- 흔한 선언적 게이트(시간창 필터)의 *보이는* 1차형.

**네가 직접.** cron 노드로 주기 발화를 흉내 내고, 트리거→실행 체인이 도는지 확인.

**리스크.**
- 스케줄러 시각 부정확 → 단조 시계 + 보정.
- 흔한 조건을 전부 코드로 숨기면 가시성 침식 → 흔한 건 보이는 노드(다음 v9의 코드 탈출구와 역할 분리).

**의존.** 선행: v4. 후행: v11(트리거가 루프 구동), v18.

---

## v9 — 인라인 코드 노드 (복잡 조건 탈출구)

**실행하면 (완결 앱).** 흐름 중간에 *코드 노드*를 박아 임의 조건으로 통과/차단·지연을 결정한다. "새벽 1시, 단 17일 제외" 같은 조건을 코드 몇 줄로. `custom` 프리셋의 실체.

**더해지는 것.**
- 인라인 코드 에디터 노드(통과/차단/지연 게이트).
- 코드 노드의 포트 선언 UX(코드여도 경계 명확, 불변식 3).
- 흔한 패턴은 보이는 선언 노드로, 코드는 진짜 임의 경우로 유도하는 넛지(가시성의 선).
- 코드 노드 격리 씨앗(v21 샌드박스가 강화).

**네가 직접.** "1시·17일 제외"를 코드 노드로 짜서 cron(v8)과 조합해 실제로 의도대로 발화/제외되는지.

**리스크.**
- 진짜 로직이 전부 코드로 숨어 그래프가 "스크립트 둘레 장식"이 됨 → 흔한 건 선언 노드로 끌어내는 넛지.
- 코드 무제한 부작용 → 격리(v21).

**의존.** 선행: v8. 후행: v11·v16·v21.

---

## v10 — 채널 backpressure 가시화

**실행하면 (완결 앱).** 엣지가 *bounded buffer*를 갖고, 버퍼가 차면 상류가 막히는 걸 캔버스에서 본다(버퍼 점유율·병목 표시). backpressure가 구조적으로 작동.

**더해지는 것.**
- 채널 bounded queue + 만수 정책(상류 블록 기본 / drop-oldest / drop-newest / 거부, 커넥션별).
- How 4프리셋(single/stream/req-resp/broadcast) 채널 동작 완성.
- fan-in/fan-out 채널 의미(복제 vs 라운드로빈, 소비자별 독립 버퍼).
- 채널 계측 → 캔버스 시각화.

**네가 직접.** 빠른 생산자 + 느린 소비자를 만들어 상류가 *실제로* 막히는지 눈으로. drop 정책별 차이 확인.

**리스크.**
- 무한 버퍼(OOM/은밀한 지연) → 기본 유한, unbounded는 명시 옵트인.
- 느린 소비자 하나가 브로드캐스트 전체 막음 → 소비자별 독립 버퍼.

**의존.** 선행: v2·v4. 후행: v11(루프 안전의 핵심), v20.

---

## v11 — 루프 + 반복 제어

**실행하면 (완결 앱).** 사이클(루프)을 캔버스에 그려 돌린다. regenerate(매 반복 새 수명주기) / retrigger(동일 영속 인스턴스) 중 선택하고, 구조적 backpressure(v10)가 폭주를 막는다. 루프가 시각화된다.

**더해지는 것.**
- 노드별 루프 행동 선언(regenerate/retrigger, v4 상태 기계 사용).
- 반복 경계·종료 조건 + 무한 루프 안전장치.
- 루프 내 backpressure 통합(폭주가 구조적으로 수렴).
- 반복 간 상태 캐리(누가 보유: 채널 vs 영속 노드) 명시.
- 빠른 루프에 시간/조건 제약을 인라인 코드(v9)로 삽입하는 패턴(단, 구조적 backpressure가 1차 방어).

**네가 직접.** 일부러 빠른 루프를 만들어 폭주시켜 보고, backpressure만으로(throttle 노드 없이) 수렴하는지 확인. 그다음 인라인 throttle을 보조로 얹어보기.

**리스크.**
- regenerate/retrigger 혼동 → 노드 속성 명시 + 시각 구분.
- "throttle 노드 박으면 됨"에만 의존 → 구조적 backpressure 기본(불변식 7).

**의존.** 선행: v8·v9·v10. 후행: v14·v18.

# Era D — 완전한 프로토콜과 합성

> 목표: 2축으로 시작한 선언형 계약을 *6축 전체*로 완성하고(추가만, D3), 폐포성·컴파일·런타임 가드로 "어떤 노드든 어디든 안전하게 붙는다"를 성숙시킨다.

## v12 — 5W1H 6축 완성 + 전체 해소 격자

**실행하면 (완결 앱).** 포트가 **6축 전부**(What/How/When/Where/Who/Why)로 선언되고, 전체 해소 격자가 연결 시 {자동호환/어댑터삽입/거부}를 결정한다. 선언형 프로토콜이 *완성*된다. v3의 2축 그래프는 그대로 작동(addition-only).

**더해지는 것.**
- When/Where/Who/Why 4축 추가(각 4프리셋 + custom). v3 격자를 6축으로 확장.
- 다축 동시 해소(한 축 거부면 전체 거부 vs 축별 어댑터 합성)의 순서·우선순위.
- 격자 전체 망라 속성 테스트("정의 안 된 칸 0").
- 마이그레이션: 기존 2축 노드를 6축 표현으로 자동 승격(D5).

**네가 직접.** 업데이트 후 *기존 그래프가 그대로 도는지* 먼저 확인(D3·D5 검증). 그다음 When/Why를 써서 "조건부 의존" 같은 걸 선언으로 표현.

**리스크.**
- 4축 추가가 직교성을 깸 → 손 교차검증.
- 6축 노출로 시각 과부하 → 흔한 정보 1차 표시, 상세 드릴다운.

**의존.** 선행: v3. 후행: v13·v14·v15.

## v13 — 변환 노드 + 어댑터 자동삽입 + 빈도 승격

**실행하면 (완결 앱).** 안 맞는 두 노드 사이에 앱이 *자동으로 변환 노드(어댑터)*를 끼워 붙인다. 같은 변환을 반복하면 "격자 규칙 후보"로 표면화. 폐포성이 작동: 빈틈은 변환 노드로 닫힌다.

**더해지는 것.**
- 변환 노드를 일반 노드로 구현(adapter/gate/throttle/collect/split — 특별 지위 없음).
- 해소 격자의 "어댑터삽입" 칸 ↔ 변환 노드 매핑, 연결 시 자동삽입.
- 자동(흔함)/수동(드묾) 경계 + **빈도 기반 승격**(반복 변환 → 규칙 후보).
- 변환 노드도 라이프사이클·경계 준수.

**네가 직접.** 일부러 안 맞는 노드를 이어 자동 어댑터가 끼는지, 모호한 칸은 거부되는지 확인. 같은 변환을 3번 해서 승격 제안이 뜨는지.

**리스크.**
- 자동삽입이 모호 변환을 끼워 조용한 오접속 → 모호 칸은 자동삽입 금지(거부/수동).
- 흔한 변환 매번 수동 재작성 → 빈도 승격으로 마찰 0 수렴.

**의존.** 선행: v12. 후행: v22.

## v14 — 구조 컴파일 + 검증 가시화

**실행하면 (완결 앱).** "컴파일" 버튼이 그래프의 *구조적* 정합성을 strict 검사해 인라인 진단을 띄운다(필수 포트 미연결, 안전장치 없는 루프, 리소스 미선언). 데이터타입은 gradual(불일치는 경고 + 런타임 가드). 불변식 6.

**더해지는 것.**
- 그래프 모델 정식화(적층/그룹, subgraph-as-node 포트 승격).
- 구조 검증기(strict-reject) + 사이클 안전 검사 + 인라인 진단.
- 컴파일 파이프라인: 매니페스트 로드→구조 컴파일→해소/어댑터 확정→데이터 gradual 검사→실행 플랜.
- 증분 컴파일(부분 변경만 재컴파일 → IDE 반응성).

**네가 직접.** 망가진 그래프(고립 노드·미연결 필수 포트·위험 루프)를 만들어 진단이 *해당 노드/엣지에 정확히* 붙는지 확인.

**리스크.**
- 구조와 데이터 검증을 섞으면 이종 노드 도메인과 싸움 → 구조만 strict.
- subgraph 포트 승격 모호 → 5W1H와 정합하게 규칙화.

**의존.** 선행: v12·v13. 후행: v15·v22.

## v15 — 경계 가드 (런타임 계약 강제 / "그래프는 거짓말 안 함")

**실행하면 (완결 앱).** 노드가 선언한 포트 계약을 *런타임에 강제*한다. 일부러 잘못 선언한 노드(선언≠행동)를 만들면 가드가 경계에서 잡고 그 노드만 격리된다(영향이 밖으로 안 샘). 신뢰의 토대.

**더해지는 것.**
- 출력/입력 경계 가드(계약 불일치 차단·보고).
- 외부(약속)/in-process(증명) 노드 가드 강도 분리 — 전자는 항상 검증, 후자는 증명분만큼 생략.
- 가드 위반 = 노드 격리(blast radius = 노드).
- 가드 비용 측정 + hot path 최적화(증명 경로 무가드).

**네가 직접.** "typed-json 낸다고 선언했는데 실제론 깨진 출력을 내는" 노드를 만들어, 가드가 잡고 *그래프가 보여준 토폴로지와 실제 흐름의 괴리를 차단*하는지 확인. (이게 v16 에이전트 신뢰의 전제.)

**리스크.**
- 가드 없으면 잘못 선언 노드가 조용히 새서 토폴로지≠실제(불변식 2·8 붕괴) → 외부 포트 가드 필수.
- 가드 과비용 → in-process 무가드 + 경계만.

**의존.** 선행: v5·v7·v12·v14. 후행: v16(에이전트 노드 봉쇄), v21.

# Era E — 에이전트 저작

> 목표: 불변식 8·9·10의 실현 — 에이전트는 stateless로 스폰돼 *노드만* 만들고, 인간은 그 부품을 배선한다. 그래프는 인간 검토용 산출물.

## v16 — 에이전트 주도 노드 생성

**실행하면 (완결 앱).** "이런 기능 노드가 필요해"를 자연어로 적으면, *stateless 에이전트 세션*이 spawn돼 node-sdk를 따라 노드를 만들고, 매니페스트 검증 + 격리 테스트(v15 가드) 통과 후 팔레트에 등록된다. 너는 그 노드를 캔버스에서 배선한다. 생성/권한 분리가 앱에서 작동.

**더해지는 것.**
- 노드 생성 요청 UI → 에이전트 세션 spawn(v7).
- SDK-가이드 생성: 에이전트에 *경계 계약만* 컨텍스트로(전체 코드베이스 아님, 불변식 3·knowledge radius).
- stateless 보장(과거 기억 없이 단일 노드, 완료 후 회수, 병렬 spawn — 불변식 10).
- 등록 파이프라인: 매니페스트 linter → 격리 실행 테스트(v15) → 팔레트 등록.

**네가 직접.** 노드 몇 개를 에이전트로 만들어 직접 배선해 돌려보기. 등록 전 격리 테스트가 *잘못 만든 노드를 거르는지* 확인. 같은 요청 병렬 투입해 간섭 없음 확인.

**리스크.**
- 에이전트가 매니페스트를 잘못 선언(약속≠행동) → v15 가드 + 등록 전 격리 테스트가 필수 관문.
- 에이전트가 경계 밖 가정(전체 이해로 회귀) → SDK 컨텍스트를 경계 계약으로 한정.

**의존.** 선행: v5·v7·v9·v15. 후행: v17·v22.

## v17 — 프로비넌스 & 되살리기

**실행하면 (완결 앱).** 모든 에이전트 생성 노드에 *세션 ID가 박히고 명세가 인라인*된다. 노드를 우클릭하면 "고치기"(명세로 재생성)와 "왜 이렇게?"(세션 기록 회상) 버튼이 있다. 서사/산물 분리(불변식 9)가 작동.

**더해지는 것.**
- 명세 인라인(5W1H 선언 + 요청 프롬프트 → 매니페스트, 재생성용).
- 세션 기록 외부 저장 + 노드엔 주소(세션 ID)만(회상용).
- 되살리기 두 경로 분리: "고치기"=재생성(새 stateless 세션), "왜?"=외부 세션 기록 회상.
- v6·v7 패널/드릴다운에서 진입점.

**네가 직접.** 만든 지 오래된 노드를 골라 "왜 이렇게?"로 당시 세션을 불러보고, "고치기"로 재생성해 *계약 호환이 유지되는지* 확인. (재생성=회상이 아님을 체감.)

**리스크.**
- ID만 박고 기록 미보존 → "왜?" 영영 불가 → 회상 원하면 기록 외부 보존 필수.
- 재생성=회상 혼동(재생성은 다른 노드 가능) → 두 경로 UI·의미 분리.

**의존.** 선행: v16. 후행: v19·v20.

## v18 — Supervisor & 자가치유

**실행하면 (완결 앱).** 노드가 죽으면 supervisor가 정책대로 되살리고(재시작 전략·backoff·상한), persistent 노드는 크래시를 생존하며, 에이전트 노드의 rate-limit을 감지해 대기 후 자동 재개한다("한도 차면 5시간 대기"의 시스템화, 구조화 신호 기반 — 불변식 11). 무인 운전 시작.

**더해지는 것.**
- supervisor 트리(one-for-one/all/rest-for-one, backoff, 격리).
- 헬스 모니터(라이프사이클·supervisor·에이전트 신호 종합).
- rate-limit/컨텍스트/멈춤 자동 복구 정책(자동 재개 상한·모드, 인간 개입 감지 시 보류).
- "마지막 메시지 재생" 기본 동작.

**네가 직접.** 노드를 무작위로 죽여 supervisor 복구를, 에이전트 한도를 흉내 내 대기→재개를 확인. 재시작 폭주가 backoff로 수렴하는지.

**리스크.**
- crash loop 자원 소모 → backoff + 상한 + 격리.
- 자동 재개 폭주/비용 폭발 → 예산·상한.
- 인간 개입 세션을 watchdog이 덮어씀 → 개입 감지 후 보류.

**의존.** 선행: v4·v7·v8·v11. 후행: v19·v24.

## v19 — 영속 & 크래시 복구

**실행하면 (완결 앱).** 앱을 껐다 켜거나 크래시해도 그래프·실행 상태가 살아남는다. 다시 열면 노드들이 *여전히 돌고 있다*. (업데이트 척추의 마이그레이션과 정합.)

**더해지는 것.**
- 런타임 상태 스냅샷(persistent 노드 상태·채널 버퍼·루프 캐리, 체크포인트 일관성).
- 재시작 복원 + 부분 실패 복구(일부 노드만 죽어도 전체 재시작 없이).
- 세션 기록(v17) 내구 저장.
- 멱등 경계로 재생 중복 억제(at-least/at-most-once 의미 명시).

**네가 직접.** 장시간 돌던 그래프를 강제 종료 후 재시작해 상태가 정확히 복원되는지, 부작용이 중복 실행 안 되는지.

**리스크.**
- 재생이 비멱등 노드를 중복 실행 → 멱등 경계/시퀀스.
- 스냅샷 비일관(레이스) → 체크포인트 일관성 프로토콜.

**의존.** 선행: v4·v17·v18 + 업데이트 척추(마이그레이션). 후행: v24.

## v20 — 관측 & 비용 추적

**실행하면 (완결 앱).** 로그·흐름 트레이스·에이전트 토큰/비용이 앱 안 대시보드에 모인다. 한 트리거가 그래프를 따라 흐른 경로·지연을 추적해서 본다.

**더해지는 것.**
- 구조화 로깅(노드/엣지/supervisor 이벤트 귀속).
- 그래프 트레이싱(트리거→경로→지연, 루프 트레이스 집계).
- 에이전트 노드(v7) 세션별 토큰/비용 집계.
- 메트릭 대시보드(처리량·지연·버퍼·실패율, v4 시각화 연동).

**네가 직접.** 복잡한 그래프를 돌리며 어디서 느려지는지·비용이 어디서 나는지 트레이스/대시보드로 짚어보기.

**리스크.**
- 관측이 hot path를 느리게 → 비동기·표본화.
- 루프 트레이스 폭발 → 집계/샘플.

**의존.** 선행: v4·v7·v18. 후행: v24.

## v21 — 보안·샌드박싱·blast-radius 강제

**실행하면 (완결 앱).** 외부/코드/에이전트 노드가 매니페스트에 *선언한 리소스만* 접근하도록 OS 레벨로 격리된다(파일시스템·네트워크·시크릿). 불변식 3을 끝까지 강제.

**더해지는 것.**
- 프로세스 샌드박싱(선언분만 허용, 가드 + OS 강제).
- 시크릿 관리(안전 주입·격리, 로그·프로비넌스 누출 금지·스크러빙).
- 네트워크 노출 제어(로컬 우선, 외부는 명시 옵트인 + 기본 차단).
- 코드/에이전트 노드(v9·v16) 격리 강화 + 권한 경계 감사 UI(인간 권한).

**네가 직접.** 노드가 선언 안 한 곳에 접근하려 할 때 막히는지, 시크릿이 로그/세션 기록에 안 새는지 직접 시도(레드팀).

**리스크.**
- 선언과 실제 접근 불일치 → 샌드박스가 선언분만 허용.
- 시크릿이 프로비넌스에 잔존 → 스크러빙·격리.

**의존.** 선행: v5·v7·v9·v15·v16. 후행: v22·v24.

## v22 — 노드 패키징·버저닝·재사용

**실행하면 (완결 앱).** 만든 노드를 패키지로 묶어 로컬 레지스트리에 보관하고, *다른 프로젝트·다른 기계*에서 재사용한다. "make once, plug anywhere"가 실제로 된다.

**더해지는 것.**
- 노드 패키지 포맷(매니페스트 + 구현 + 인라인 명세[v17] + 의존).
- 버저닝·호환(프로토콜 addition-only 규칙[D3]을 패키지 semver로).
- 로컬 레지스트리(보관·검색·재사용, 팔레트 연동).
- 의존 해소·버전 잠금 + 재현성(언어 무관 노드 포함, v5).

**네가 직접.** 한 프로젝트에서 만든 노드를 다른 프로젝트에 가져와 *그대로 작동하는지* 확인. "어디선 되고 어디선 안 됨"이 없는지.

**리스크.**
- 환경 의존 누설 → 리소스/의존을 매니페스트에 명시 + 샌드박스 격리.
- 버전 변경이 기존 그래프를 조용히 깸 → semver + 컴파일 시 비호환 경고(v14) + 마이그레이션(D5).

**의존.** 선행: v5·v13·v14·v16·v21. 후행: v24.

## v23 — 멀티플렉서 성숙

**실행하면 (완결 앱).** 다수의 라이브 터미널·에이전트 노드를 타일/탭으로 동시에 보고 관리한다. v6의 씨앗이 본격 멀티플렉서 경험으로 성숙(세션 카드·라이브 peek·캔버스↔패널 양방향 하이라이트).

**더해지는 것.**
- 멀티 세션 타일/탭 뷰 + 세션 카드(상태·빠른 동작).
- 라이브 peek·검색·파일 미리보기.
- 비활성 패널 렌더 보류(성능).
- 세션-노드 양방향 하이라이트 강화.

**네가 직접.** 여러 에이전트/터미널 노드를 동시에 띄워두고 한 화면에서 관리해보기(실사용 부하).

**리스크.**
- 다수 동시 렌더 성능 → 비활성 보류.
- 입력 충돌(패널 키 + 주입) → 모드 표시·큐 직렬화.

**의존.** 선행: v6·v7·v18. 후행: v24.

## v24 — 통합 하드닝 · eval 하니스 · 릴리스 성숙

**실행하면 (완결 앱).** 전 시스템이 통합 시나리오를 관통하고, 회귀를 막는 eval이 돌고, 너는 *이 앱으로 이 앱의 일부 워크플로(빌드/배포/eval)*를 운영한다. "무인 신뢰"의 최종 바.

**더해지는 것.**
- e2e 통합 시나리오("1시·17일 제외 cron→에이전트 노드→변환→출력", "rate-limit 대기 루프")를 전 스택 관통.
- eval 하니스(노드 생성 품질·해소 격자 정확성·가드 강제 회귀 측정) — CLI(D6)로도 헤드리스 실행.
- **불변식 감사**(제품 11 + 배포 6이 통합 환경에서 유지되는지 — 특히 "그래프는 거짓말 안 함", "모든 버전 실행됨").
- 릴리스 성숙(자동 업데이트·서명·롤백·마이그레이션의 최종 하드닝).
- 도그푸딩: 이 IDE로 이 IDE의 파이프라인 운영.

**네가 직접.** 대화에서 나온 실제 케이스를 전부 앱에서 재현. 장시간 무인 운전 + 불변식 감사 통과를 릴리스 기준으로.

**리스크.**
- 통합에서야 드러나는 교차 의미 불일치(라이프사이클×루프×backpressure×가드) → 불변식 감사를 1급 관문으로.
- "데모는 되는데 무인 신뢰 불가" → v18 장시간 운전 + 감사 통과가 릴리스 기준.

**의존.** 선행: 사실상 전체. 후행: 없음(릴리스). 이후는 운영·반복.

## 4. 임계 경로와 누적 규칙

- **임계 점화: v1.** "실행되고 자기 업데이트되는 최소 앱"이 서는 순간 도그푸딩 루프가 점화되고, 이후 모든 버전이 업데이트로 흘러든다. v1이 늦으면 *전체가 늦는다*.
- **최소 쓸모: v5.** 네 실제 스크립트가 노드로 도는 순간부터 앱이 장난감을 넘어선다.
- **신뢰의 분기: v15.** "그래프는 거짓말 안 함"이 런타임으로 강제되는 순간, 에이전트 생성 노드(v16)를 믿고 배선할 수 있다.
- **무인 신뢰: v18~v19.** 자가치유 + 영속이 서면 "자러 가도 되는" 상태에 진입.
- **누적 규칙:** 모든 버전은 이전 전부를 포함하고 퇴행하지 않는다(D1). 업데이트 척추(§2)는 v1~v24를 관통한다.

## 5. 이전 문서 대비 무엇이 바뀌었나

내용(11개 제품 불변식·각 서브시스템의 위험·교차 의존)은 보존됐다. **구조만** 수평 레이어 → 수직 버전으로 뒤집혔다. 옛 Part↔새 Era 대략 대응:

- 옛 Part I(프로토콜)·II(커널) → v1~v4 + v10·v12에 *분산*(전부 미리 완성하지 않고, 실행되는 슬라이스로 쪼개 점증).
- 옛 Part III(호스트) → Era B(v5~v7).
- 옛 Part IV(합성/컴파일) → Era C·D(v8~v15).
- 옛 Part V(IDE) → 별도 era가 아님. **IDE는 v1부터 존재**하고 매 버전 두꺼워진다(D2).
- 옛 Part VI(에이전트 저작) → Era E(v16~v17).
- 옛 Part VII(하드닝/운영) → Era F·G(v18~v24).

핵심 신규: **업데이트 & 마이그레이션 척추(§2)**, **addition-only 프로토콜 성장(D3)**, **GUI 일급/CLI 보조(D6)**. 이 셋이 "각 버전이 실행되는 완결 앱이고, 업데이트로 배포되며, 내 작업을 보존한다"를 성립시킨다.

## 6. 계속 주시할 항목

대화에서 결론졌으나 구현 내내 후퇴 압력이 올 지점들.

1. **모든 버전이 실제로 실행되는가(D1).** "이번엔 헤드리스로만 쌓고 다음에 합치자"는 유혹 → 매 버전 실행·도그푸딩이 곧 검증.
2. **업데이트가 작업을 보존하는가(D5).** 포맷·스키마 변경 시 마이그레이션 누락 → "업데이트 후 기존 그래프 작동"을 매 버전 첫 확인 항목으로.
3. **프로토콜은 추가로만(D3).** 배포된 프리셋 의미를 "조금만" 바꾸려는 유혹 → 추가만, 변경 금지.
4. **데이터타입 gradual·구조 strict(불변식 6).** 데이터를 strict로 끌어올리고 싶은 반복 압력 → 외부 노드는 "약속"이라는 사실이 이를 금함.
5. **backpressure 구조적 1차(불변식 7).** "throttle 노드 박으면 됨"으로 후퇴 금지.
6. **에이전트 구동은 구조화 스트리밍(불변식 11).** 스크레이프가 빠르다는 유혹에 후퇴 금지.
7. **킬러 경로 하나.** 최대 일반성을 추구하되, 명백히 제일 잘하는 워크플로 하나(무인 에이전트 오케스트레이션)를 함께. 일반 substrate ≠ 무목적.
